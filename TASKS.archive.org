** DONE [#D] Fix compact widget not updating after selection toggle :extension:
:PROPERTIES:
:ID: 8e1f3c72-4a9d-4b15-a6c0-2d7e9f1b3c48
:END:
When the user toggles =:selected:= inside the =/tasks= overlay, the
compact widget above the editor (the "editor overlay") should reflect
the new selection immediately after the overlay closes. Two related
issues prevented this:

1. The file watcher fires ~150ms after =save()= and calls =refreshTaskUi=,
   which tried to *create* the compact widget while the expanded overlay
   was still visible — producing visual artifacts or a stale-state race.
2. After the overlay closed, =refreshTaskUi= read from disk, racing with
   the async fire-and-forget =save()= call from inside the overlay.

Fix (=index.ts=):
- Added =isOverlayActive= module-level flag, set =true= at the top of each
  =/tasks= overlay loop iteration and =false= after the loop exits.
- =refreshTaskUi= (called from the file-watcher path) skips
  =syncCompactWidget= when =isOverlayActive= is set.
- After the loop exits, call =syncCompactWidget(ctx, tasks)= immediately
  using the in-memory task state before calling =refreshTaskUi=.
** DONE [#D] Fix overlay cursor starting at top-level task :extension:
:PROPERTIES:
:ID: 3b7d5e91-6f2a-4c08-8e4b-1a9c0d7f2e35
:END:
When the =/tasks= overlay opens, the cursor should land on the selected
task (the one carrying =:selected:=). =focusSelectedTask()= is called in
the constructor and searches =this.rows= for the =isSelectedTask= row —
but the row was absent when the selected task was a =planChild= of a
DONE/CANCELLED parent.

Root cause: =applyDefaultCollapseView()= contained a third condition that
collapsed tasks in =keepVisible= (ancestors of the selected task) if they
were DONE/CANCELLED and not the selected task itself. This contradicted
the purpose of =keepVisible=: tasks on the path to the selected task must
remain expanded. With the parent collapsed, the planChild was not walked
in =rebuildRows()=, =focusSelectedTask()= returned =-1=, and =cursor=
stayed at 0 (the first top-level task).

Fix (=overlay.ts=): removed the erroneous third condition. The two
remaining conditions correctly express the intended rules:
- No selection → collapse everything with children.
- With selection → collapse only tasks not in =keepVisible=, covering
  completed non-ancestor tasks automatically.

Deferred follow-up. The current bindings (=, K {s,i,I,p,t}=) work but the
implementation is ugly:
- Capital =K= is not mnemonic for "tasks".

  =+lang/org='s late binding.

Acceptance criteria:
- A keybinding scheme that does not require unbinding existing Spacemacs
  org-layer bindings, OR
- A clearly justified scheme under a stable prefix that is unlikely to be
  reclaimed by upstream Spacemacs.

Candidate approaches to evaluate:
- Move to a global leader (e.g. =SPC a t= for "applications > tasks") so
  bindings don't compete with org-mode prefixes at all.
- Use a special-character prefix (e.g. =, != or =, ;=) that Spacemacs does
  not bind in any layer.
- Nest under a sub-prefix already declared by Spacemacs (requires surveying
  which prefixes are designated for layer extension).
- Accept the current scheme as intentional and document the trade-off.

[2026-04-27 Sun] Adopted =, ;= as the prefix. =;= is free in the Spacemacs
org layer, reachable without Shift, and requires no unbinding of existing
bindings. The =with-eval-after-load 'org= timing workaround was removed
entirely. Candidate approaches evaluated and rejected: =SPC a t= (global
leader, too many keystrokes for frequent use), single free capital letters
(non-mnemonic). =;= is non-mnemonic too but is the most ergonomic free key.

Not blocking v1. Revisit when the package is stable enough that a breaking
key change is acceptable, or before sharing the package beyond personal use.
