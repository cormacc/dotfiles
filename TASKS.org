Phase 2 of the home-network-infrastructure roadmap (see `~/dotfiles/design/home-network-infrastructure.org`). Introduce Caddy on strix as the LAN-facing reverse proxy with stable hostnames under `home.arpa`, so user-facing URLs decouple from host+port pairs.

Hostname plan (per existing decision, RFC 8375):
- `chat.home.arpa` → Open WebUI (currently strix:8080)
- `search.home.arpa` → Vane (currently strix:3001)
- `lemonade.home.arpa` → Lemonade OpenAI-compatible API (currently strix:13305)
- SearXNG remains internal (no public Caddy vhost; Vane / Open WebUI / pi continue hitting it via `host.docker.internal` / `127.0.0.1` / `strix:8888` respectively).

Worth promoting to its own change-record via planner — multiple design decisions need pinning before code:

Design questions for the planner pass:
- DNS source for `*.home.arpa`: UniFi DNS override vs Pi-hole/AdGuard vs `services.resolved` host records. Whichever resolves on every LAN client without per-device config.
- TLS strategy: skip TLS for LAN (HTTP-only) vs internal CA + trust on devices vs ACME DNS-01 against a real domain. Probably HTTP for Phase 2, defer TLS.
- Should backends bind 127.0.0.1 once Caddy fronts them (per existing "Prefer localhost-only backends" decision)? Open WebUI and Vane currently bind 0.0.0.0 + openFirewall. Tightening should be the same NixOS pass as adding Caddy.
- Caddy NixOS module: `services.caddy` is in nixpkgs; verify it covers what's needed.

Acceptance shape (refine in planner pass):
- All three hostnames resolve on LAN clients without per-device hosts-file edits.
- `curl http://chat.home.arpa` reaches Open WebUI.
- `curl http://search.home.arpa` reaches Vane.
- `curl http://lemonade.home.arpa/api/v1/models` works for the OpenAI-API clients (verify they accept the new base URL — pi config, etc.).
- Backends no longer LAN-reachable on their raw host:port (or this is deferred to a followup; flag explicitly).

Expose a curated subset of home services to the public internet, using strix Caddy as the TLS terminator and UniFi as the WAN gateway with port-forward to Caddy. Blocked by the LAN Caddy task (`0e5c627e-1297-4609-8d11-5081c6dae485`) — needs Caddy already running internally before any public path is added.

Per-service exposure policy (see `~/dotfiles/design/home-network-infrastructure.org` § "Per-service exposure model"):

| Service    | Public OK? | Reason                                              |
|------------+------------+-----------------------------------------------------|
| Open WebUI | YES        | Has auth; designed for external use                 |
| Vane       | NO         | No auth by design                                   |
| SearXNG    | NO         | Internal backend; no value to expose                |
| Lemonade   | NO         | Compute-abuse risk; raw LLM API                     |

So the initial public surface is **Open WebUI only**.

Design questions for the planner pass:
- Public DNS: separate public hostname (e.g. `chat.<somedomain>`), not `chat.home.arpa`. Where does the apex domain live (e.g. existing Cloudflare / registrar)?
- TLS: ACME via HTTP-01 (UniFi forwards :80 + :443) or DNS-01 against the public domain. DNS-01 keeps :80 closed.
- UniFi port forward: WAN :443 → strix Caddy. Verify UniFi firmware version + recent config patterns.
- Caddy multi-vhost: same Caddy serves `chat.home.arpa` (LAN HTTP) and `chat.<somedomain>` (public HTTPS) for the same Open WebUI backend.
- Open WebUI hardening before exposing: forced strong passwords, disable signup, audit log review, rate-limit at Caddy, geo-block via Caddy if desired.
- Threat model: brute-force on the login form is the dominant risk. Caddy-level rate-limit and fail2ban-equivalent before going live.
- DDNS: dynamic public IP handling if the home WAN IP changes (UniFi has built-in DDNS options or a Cloudflare API hook).

Acceptance shape (refine in planner pass):
- `https://chat.<publicdomain>/` reaches Open WebUI login from any internet client.
- TLS cert is auto-renewing.
- Vane / SearXNG / Lemonade are NOT exposed (negative test from off-tailnet, off-LAN device).
- UniFi forward + Caddy config both checked into NixOS where possible (UniFi probably manual unless using UniFi's API).

Should be promoted to its own change-record via planner once the LAN Caddy task is done and Open WebUI exposure is the current priority.
