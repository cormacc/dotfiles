* Overview
My homefolder configuration, managed as a Nix flake using Home Manager.
Formerly managed using the [[https://github.com/cormacc/dotmodule][dotmodule]] gem.

The flake covers three deployment shapes:
- *Home Manager* (standalone) on Arch / hybrid Linux installs and on macOS via nix-darwin -- this is the primary daily-driver path.
- *NixOS* system configurations for several hosts (=xps15=, =strix=, =t580=, =nas=). =xps15= is the canonical pattern (system config + standalone home-manager applied separately).
- *nix-darwin* system configuration for =Cormacs-MacBook-Air= (with home-manager included as a darwin module).

The golden path for development machines is currently Arch Linux + nix + home-manager rather than full NixOS, due to difficulties packaging some toolchains required for embedded development, specifically:
- ZephyrOS toolchain (incorporates some dynamically linked binaries)
- Microchip compilers (dynamically linked binaries + license to be applied post-installation)

Progress on the Microchip side lives under [[./microchip/]] and the [[https://github.com/cormacc/nix-microchip][nix-microchip]] flake input.

* Repository layout / architecture

** Flake outputs
[[./flake.nix]] defines three output groups:
- =nixosConfigurations= -- one per host: =xps15=, =strix=, =t580=, =nas= (currently the odroid-h4).
- =homeConfigurations= -- =default= (full Linux workstation, imports [[./home.nix]]) and =minimal= (server / WSL, imports [[./home-linux.nix]]).
- =darwinConfigurations."Cormacs-MacBook-Air"= -- macOS system config with home-manager bundled as a darwin module.

 =allowUnfree = true= is set globally; =--impure= is required on all builds (the flake reads identity env vars).

** NixOS module layering
[[./nixos-workstation.nix]] imports [[./nixos-base.nix]] and adds workstation concerns (audio, display, sway, hyprland). Host-specific configs live in =hosts/<hostname>/=. Optional mixins:
- [[./nixos-nvidia.nix]] -- GPU drivers (set =dotfiles.nvidia.legacy = true= for pre-Turing cards)
- [[./nixos-gaming.nix]] -- Steam and gaming packages
- [[./nixos-llm.nix]] -- ollama / LLM setup
- [[./nixos-server.nix]] -- server-only config
- [[./nixos-boot-default.nix]] -- standard systemd-boot setup
- [[./nixos-bootstrap.nix]] -- minimal config dropped onto a fresh install via [[./bootstrap.sh]]

** Home Manager module layering
#+begin_example
home-core.nix  ->  shell/shell.nix
      ^
home-linux.nix
      ^
home.nix (full Linux workstation, adds:)
      |-- editors/editors.nix    (emacs configs: corgi, doom, spacemacs)
      |-- dev/dev.nix            (dev tooling, clojure)
      |-- dev/dev-linux.nix
      |-- desktop/{web,audio,office}.nix
      |-- wayland/wayland.nix    (foot, rofi, fontconfig)
      |-- wayland/sway/sway.nix
      |-- wayland/hypr/hypr.nix
      |-- nmd/nmd.nix            (OneDrive / work tooling)
      \-- agents.nix             (pi + claude-code)

home-darwin.nix  ->  home-core.nix  (macOS, adds: editors, dev, agents)
#+end_example

[[./home-core.nix]] owns shell config, direnv, git, ssh, vim, fonts, XDG, and defines the =hms= / =nos= / =drs= shell aliases.

** Environment variables consumed on first apply
The flake uses =builtins.getEnv= for identity and a couple of secrets, hence =--impure=. Persisted to the shell session after first apply, so you only need to set them once per installation:
- =NAME=, =EMAIL=, =USER=, =HOME= -- identity and paths. =EMAIL= is the default git =user.email= (typically the work address).
- =EMAIL_OSS= -- personal / OSS git =user.email=. Scoped to =~/dev/= repos via an =includeIf= so personal commits don't leak the work address. Optional: when unset it falls back to =$EMAIL=, making it a no-op for single-identity setups.
- =GITLAB= -- self-hosted gitlab TLD for SSH config (optional)
- =OPENAI_API_KEY=, =ANTHROPIC_API_KEY= -- propagated to the shell env (optional)

See [[#set-git-name-and-e-mail][Set Git name and e-mail]] for the actual =export= commands.

* The dotagents submodule

** What it is
Agent skills, pi extensions, prompt templates, the pi-side =AGENTS.md=,
and the =agent-org-memory= pi package live in a sibling repo at
[[https://github.com/cormacc/dotagents][github:cormacc/dotagents]] — see that repo for the full inventory of agent
skills and configuration. This repo registers it as a git submodule under
[[file:./agents/][=agents/=]].

Why a submodule rather than a flake input? Editability. Home Manager
points =mkOutOfStoreSymlink=s at the live submodule working tree, so
editable changes (skills, extensions, prompts) reload via pi's =/reload=
without a Home Manager switch. A flake input would resolve to an
immutable Nix store path and lose that property.

** Bootstrap
#+begin_src shell
  # Fresh clone
  git clone --recurse-submodules https://github.com/cormacc/dotfiles ~/dotfiles

  # Already cloned without --recurse-submodules
  git -C ~/dotfiles submodule update --init --recursive
#+end_src
[[./agents.nix]] runs a fail-fast =home.activation= guard that refuses
to activate when =agents/package.json= is missing, with the same
instruction in the error message.

** Update workflow
#+begin_src shell
  # Pull the latest dotagents commit, leaving you on a detached HEAD
  git -C ~/dotfiles/agents fetch origin
  git -C ~/dotfiles/agents checkout origin/main
  # Or shorthand:
  git -C ~/dotfiles submodule update --remote agents

  # Commit the new submodule pointer in this dotfiles repo
  git -C ~/dotfiles add agents
  git -C ~/dotfiles commit -m "chore(agents): bump dotagents pointer"
#+end_src

** Editing in place
Work inside =agents/= directly, push to the dotagents repo, then bump the
dotfiles submodule pointer (see *Update workflow*) so other checkouts
pick up the new revision. Pi sessions on the same machine see edits
immediately because the install paths are symlinks into the live working
tree.

** Releasing dotagents
Dotagents tags are advisory -- the dotfiles submodule pointer is the
real pin. To tag a meaningful slice of work:
#+begin_src shell
  cd ~/dotfiles/agents
  git tag -a v0.2.0 -m "agent-org-memory + foo extension"
  git push origin v0.2.0
#+end_src
Then bump the dotfiles submodule pointer (see *Update workflow*).

* The pi-settings clean filter

[[file:./agents/pi/settings.json]] (inside the dotagents submodule) is a
single source of truth for two very different concerns: a stable list
of installed pi packages (=packages= field) and volatile runtime
preferences (=defaultProvider=, =defaultModel=, =lastChangelogVersion=)
that pi rewrites on every =/model= swap or version bump. Without
intervention the volatile fields show up as repo churn.

A git clean filter strips them when staging while leaving the working
tree untouched. Pi keeps writing through unchanged; git only ever sees
the durable subset.

Because =pi/settings.json= lives inside the submodule, the filter must
be registered against the *submodule's* =.git/config=, not the parent
dotfiles repo's.

** First-time setup
After cloning, register the filter once from inside the submodule:
#+begin_src shell
  ~/dotfiles/agents/install-git-filter.sh
#+end_src
The script writes three entries into =agents/.git/config= under
=[filter "pi-settings"]= and is idempotent. If =jq= is missing it
fails fast with a clear error.

The submodule's [[file:./agents/.gitattributes]] entry binds the filter
to =pi/settings.json= so subsequent =git add= / =git commit=
invocations inside =agents/= honour it automatically.

** Verifying it works
#+begin_src shell
  cd ~/dotfiles/agents

  # Edit a runtime field directly:
  jq '.defaultModel = "some-other-model"' \
    pi/settings.json > /tmp/settings.json && \
    mv /tmp/settings.json pi/settings.json

  # git status should be clean (filter elides the change):
  git status pi/settings.json

  # Add a package to the persisted list — git status DOES report this:
  jq '.packages += ["npm:foo"]' \
    pi/settings.json > /tmp/settings.json && \
    mv /tmp/settings.json pi/settings.json
  git diff pi/settings.json
#+end_src

** Adjusting the field list
Edit the =clean= command in [[file:./agents/install-git-filter.sh]] and
re-run the install script. The =jq 'del(...)'= expression takes
comma-separated paths.

* Known issues / limitations

OpenGL applications under Home Manager on non-NixOS used to require
[[https://github.com/nix-community/nixGL][NixGL]]; home-manager's =targets.genericLinux.enable= now seems to handle
this transparently (see the [[https://nix-community.github.io/home-manager/index.xhtml#sec-usage-gpu-non-nixos][HM docs]]). Not currently re-verified on a
non-NixOS install.

* Provisioning a new account/installation

The dotagents submodule must be initialised before any =home-manager
switch= -- see [[#bootstrap][The dotagents submodule § Bootstrap]]. Then follow the
platform-specific section below.

** Non-NixOS
Install Nix via the [[https://zero-to-nix.com/start/install][Determinate Systems installer]] (enables flakes
by default; sets up daemon sourcing in your shell init):
#+begin_src shell
  curl --proto '=https' --tlsv1.2 -sSf -L https://install.determinate.systems/nix | sh -s -- install
#+end_src

Then install Home Manager per the [[https://nix-community.github.io/home-manager/index.html#ch-installation][official docs]]:
#+begin_src shell
  nix-channel --add https://github.com/nix-community/home-manager/archive/master.tar.gz home-manager
  nix-channel --update
  nix-shell '<home-manager>' -A install
#+end_src

** NixOS
NixOS allows home-manager to be bundled as a system module, but the
preferred pattern (demonstrated by =xps15=) is to keep the NixOS config
minimal and apply home-manager standalone, since user-level config
churns much faster than the OS-level.

*** Install NixOS
Use the official ISO installer. After install, =/etc/nixos/configuration.nix= and =hardware-configuration.nix= are generated for the host.

*** Amend NixOS config to allow flakes, install git etc.
Enable flakes and add =git= so that the repo can be cloned and the flake evaluated:
#+begin_src nix
  nix.settings.experimental-features = [ "nix-command" "flakes" ];
  environment.systemPackages = with pkgs; [ git ];
#+end_src
Then rebuild with =sudo nixos-rebuild switch=.

[[./nixos-bootstrap.nix]] is a checked-in minimal config that does roughly this. [[./bootstrap.sh]] copies it over =/etc/nixos/configuration.nix= and rebuilds.

*** Integrate host hardware configuration and apply NixOS config from flake
For a brand new host:
1. Create =hosts/<hostname>/= and copy =/etc/nixos/hardware-configuration.nix= into it.
2. (Optionally) add a =hosts/<hostname>/nixos-configuration.nix= for host-specific overrides (see =hosts/xps15/= for an example).
3. Add a new =nixosConfigurations.<hostname>= entry to [[./flake.nix]] importing the relevant mixins (workstation / server / nvidia / gaming etc.).
4. Apply:
   #+begin_src shell
     sudo nixos-rebuild switch --flake .#<hostname> --impure
   #+end_src

After the first successful apply, the =nos= shell alias (defined in [[./home-core.nix]]) auto-detects the hostname for subsequent rebuilds.

** macOS (nix-darwin)
Install Nix using the [[https://zero-to-nix.com/start/install][Determinate Systems installer]] as per the non-NixOS section.

Install nix-darwin per the [[https://github.com/nix-darwin/nix-darwin][official instructions]], then apply the flake:
#+begin_src shell
  export NAME="<Your Name>"
  export EMAIL="<your.email>@<domain.com>"
  sudo darwin-rebuild switch --flake "$HOME/dotfiles#Cormacs-MacBook-Air" --impure
#+end_src
The darwin config bundles home-manager as a module (using the =minimal= cfgName, plus editors / dev / agents) -- so a separate =home-manager switch= is not required on macOS. The =drs= alias is wired up after the first apply.

** Setup your home environment

*** Set Git name and e-mail
The flake reads identity from environment variables on initial application via =builtins.getEnv= (hence the =--impure= flag on all rebuild commands). These are propagated into the shell config for future re-applications -- you only need to set them once per installation.

#+begin_src shell
  export NAME="<Your Name>"
  export EMAIL="<your.email>@<domain.com>"
  # Optional -- personal / OSS git email, scoped to ~/dev/ repos via a
  # git `includeIf`. Falls back to $EMAIL when unset.
  export EMAIL_OSS="<personal.email>@<domain.com>"
  # Optional -- self-hosted gitlab TLD for SSH host alias
  export GITLAB="<gitlab.example.com>"
  # Optional -- propagated to your shell env after first apply
  export OPENAI_API_KEY="<...>"
  export ANTHROPIC_API_KEY="<...>"
#+end_src
=USER= and =HOME= are also consumed but set automatically at login.

*** SSH keys / host aliases
On my work machines, I typically setup two ssh keys -- one for my work identity/e-mail, and the other for my personal e-mail address which I use for open source contributions etc. (github and gitlab.com).
The primary key is the work key, located at [[~/.ssh/id_ed25519]].
In order to use the personal key automatically for relevant hosts, a couple of aliases are setup in [[./home-core.nix]]
An additional alias has been defined for administration of a self-hosted gitlab instance on AWS. This requires the host tld to be specified as an environment variable (=GITLAB=) when initially applying the flake.

*** Apply the Home Manager flake
Minimal (server / WSL, no desktop):
#+begin_src shell
  export NIXPKGS_ALLOW_UNFREE=1
  home-manager switch --flake './dotfiles#minimal' --impure -b backup
#+end_src

Workstation (full desktop):
#+begin_src shell
  export NIXPKGS_ALLOW_UNFREE=1
  home-manager switch --flake './dotfiles#default' --impure -b backup
#+end_src

This first apply defines the =hms= / =nos= / =drs= shell aliases (see [[#shell-aliases][Maintenance § Shell aliases]]).

** Post-installation activities

*** Generate a unique SSH key for this installation
Use the newer ED25519 algorithm (per [[https://docs.github.com/en/authentication/connecting-to-github-with-ssh/generating-a-new-ssh-key-and-adding-it-to-the-ssh-agent][Github instructions]]).
#+begin_src shell
  ssh-keygen -t ed25519 -C "$EMAIL"
#+end_src

Then add the new key via Gitlab settings.


*** Hybrid/arch linux additional setup steps

**** Printing (CUPS, Avahi etc.)
Install cups, hplip, hplip-plugin
Follow instructions on avahi setup on [[https://wiki.archlinux.org/title/Avahi][arch wiki]] for network printer discovery to work.

**** Microchip MPLAB-X
I've made some progress (building on other people's work) to getting a nix-based setup for microchip development. However the licensed compiler is still a stumbling block. For now, installing MPLAB-X under Arch linux via the AUR.

MPLAB-X requires some funky setup -- see [[./microchip/README.org]] for instructions and further info.

* Maintenance

** Shell aliases
Defined in [[./home-core.nix]] and available after the first successful apply:
| Alias | Expands to |
|-------+------------|
| =hms= | =home-manager switch --flake './dotfiles#<cfgName>' --impure= |
| =hmb= | =home-manager build --flake './dotfiles#<cfgName>' --impure= |
| =nos= | =sudo nixos-rebuild switch --flake './dotfiles#$(hostname)' --impure= |
| =nob= | =nixos-rebuild build --flake './dotfiles#$(hostname)' --impure= |
| =drs= | =sudo darwin-rebuild switch --flake './dotfiles#<host>' --impure= |

** Updating packages
=flake.lock= pins nixpkgs for reproducibility. To update:
#+begin_src shell
  nix flake update
  hms   # or nos / drs depending on host
#+end_src

** Managing generations
A new generation is built on every =home-manager switch= (or =nixos-rebuild switch=).

*Home Manager:*
#+begin_src shell
  home-manager generations                       # list
  home-manager remove-generations $(seq 1 18)    # delete by id
#+end_src
HM has no built-in rollback. To activate an older generation, run the
=activate= script inside the generation's =/nix/store= path (shown by
=home-manager generations=) and optionally delete later generations.

*NixOS:*
#+begin_src shell
  nix-collect-garbage --delete-old   # generations >1 week old
  sudo nix-collect-garbage -d        # full sweep, also frees /boot kernels on next rebuild
#+end_src

** Updating NixOS
For flake-based hosts:
1. =nix flake update= to bump locked inputs.
2. Optionally =sudo nixos-rebuild test --flake .#<hostname> --impure= to build without activating.
3. =nos= (or =sudo nixos-rebuild switch --flake .#<hostname> --impure= explicitly) to activate.
4. Optionally =sudo nix-collect-garbage -d= afterwards.
