* Intent

Stand up one shared web-search stack on the =strix= NixOS host so every workstation — on LAN or over Tailscale — has consistent search-augmented chat (Open WebUI), a Perplexity-style human UI (Vane), and a raw search API for agents (SearXNG via MCP). No per-workstation instances, no Darwin support.

* Summary

*Level:* production · *Tests:* smoke · *Docs:* inline

Single-node topology on =strix= (Framework Desktop, NixOS, =nix-amd-ai= + Lemonade). Search backbone is a local SearXNG with JSON output and Wolfram Alpha enabled. Two human consumers (Open WebUI for chat, Vane for Perplexity-style answers) both call SearXNG and Lemonade's OpenAI endpoint. Agents skip Vane and hit SearXNG directly via the =mcp-searxng= MCP server wired into this dotagents repo. Off-net access is via Tailscale MagicDNS; the home firewall is the trust boundary.

** Scope
*** In scope
- =services.searxng= NixOS module on strix with JSON format + Wolfram + curated engines.
- =services.open-webui= web-search settings pointed at the local SearXNG.
- =virtualisation.oci-containers.containers.vane= on strix, wired to SearXNG + Lemonade.
- =mcp-searxng= entry in this repo's =mcp.json= (source of truth for =~/.config/mcp/mcp.json=).
- Tailscale/firewall reachability check for SearXNG, Open WebUI, Vane.
- End-to-end smoke from one off-net workstation.

*** Out of scope
- Per-workstation Vane/SearXNG instances.
- Darwin support (=strix= is Linux-only; no MacBook fallback).
- Auth in front of any of the three services (LAN + tailnet are trusted).
- Subscription-OAuth fallbacks (Anthropic banned them Feb 2026; off-net path is Tailscale).
- Caddy / reverse proxy in front of these services (deferred; Phase-1 strix UIs are already raw-LAN).
- Replacing Open WebUI's bundled search with Vane.

** Decisions
- *SearXNG, not standalone Wolfram or Brave API* :: single source of truth that all three consumers share; the report flagged Vane requires the SearXNG Wolfram engine specifically.
- *Agents hit SearXNG directly, not Vane* :: pi tooling wants raw result lists, not LLM summaries. Avoids a double-hop through a second model and keeps token cost predictable. Vane stays human-only.
- *Vane via =oci-containers=* :: upstream ships Docker only; researcher confirmed no native Nix package. =oci-containers= is the existing dotfiles idiom for Docker-only services.
- *Tailscale MagicDNS hostnames in =mcp.json=* :: same string resolves on-LAN and off-net, so the MCP server config is portable across workstations without conditional logic.
- Rejected:
  - *Per-workstation Vane instances for off-net use* :: Tailscale already solves reachability; running N Vane copies multiplies state and config drift.
  - *Standalone Perplexica/Morphic/Farfalle/LLocalSearch* :: Vane (Perplexica fork) is the active maintained path per the report.
  - *MCP via Vane (=perplexica-mcp=)* :: forces every agent search through an LLM summary step we do not want.

** Shipped
- SearXNG on =strix:8888= via =services.searx= (JSON + Wolfram + curated engines, =secret_key= from out-of-store env file).
- Open WebUI on =strix:8080= wired to local SearXNG + Lemonade, with an =ExecStartPre= bootstrap re-applying PersistentConfig web-search settings on every boot.
- Vane on =strix:3001= via =oci-containers= (=itzcrazykns1337/vane:slim-v1.12.2=), with an =ExecStartPre= bootstrap that uses =jq= to keep =search.searxngURL= in =/var/lib/vane/config.json= aligned with =SEARXNG_API_URL=.
- =mcp-searxng= registered in dotagents =mcp.json= pointing at =http://strix:8888= via Tailscale MagicDNS — same string resolves on LAN and (once Tailscale lands on strix) over tailnet.
- LAN end-to-end verified: SearXNG curl, Vane =/api/chat= smoke (Lemonade chat + Transformers embeddings + Wolfram-engine result), pi MCP =searxng_web_search= call.

** Gotchas
- /Tools with =PersistentConfig=-style state need an idempotent =ExecStartPre= bootstrap./ Open WebUI and Vane both persist runtime state and prefer it over env vars after first boot. Declarative env-only configuration silently loses to admin-UI or setup-wizard state. Pair env with a re-apply step (Open WebUI: PATCH admin API; Vane: =jq= on =config.json=). Expect the same pattern for similarly-structured services.
- /Open WebUI 0.9.5 renamed the web-search env vars./ Stale names =ENABLE_RAG_WEB_SEARCH= / =RAG_WEB_SEARCH_ENGINE= are silently ignored. Current names: =ENABLE_WEB_SEARCH=, =WEB_SEARCH_ENGINE=, =SEARXNG_QUERY_URL=. Verify against =config.py= in the deployed package version when wiring.
- /Vane's setup wizard wrote a self-referential =searxngURL=./ The form pre-filled =http://strix.internal:3001= (Vane's own URL) into =/var/lib/vane/config.json=. Vane's =initializeFromEnv()= only seeds from =SEARXNG_API_URL= when the persisted value is falsy, so wizard fat-fingers win permanently. The =ExecStartPre= bootstrap above makes this idempotent.
- /SearXNG rejects the module default =secret_key=./ =services.searx= defaults to =server.secret_key = "ultrasecretkey"=, which SearXNG refuses to start with. Generate a real secret at activation and pass via =services.searx.environmentFile= (out of Nix store), referenced as =$SEARX_SECRET_KEY= in settings.
- /Pinned nixpkgs uses =services.searx=, not =services.searxng=./ They alias to the same package but the option path differs per channel. Verify with =nix eval= against the pinned commit before writing the host config.
- /Lemonade port =13305= is single-sourced./ Defined as =lemonadePort= in =hosts/strix/nixos-configuration.nix= and consumed via =${toString lemonadePort}= by both Open WebUI and Vane. Don't hard-code the port downstream.
