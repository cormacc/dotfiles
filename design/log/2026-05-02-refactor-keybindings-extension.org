* Context

The =keybindings= pi extension under =agents/pi/extensions/keybindings/=
has grown into two distinct responsibilities living in one ~3000-line
=index.ts=:

1. *Leader-menu (which-key style)* — discoverable Space and =,= leader chords,
   the cross-extension =suggestKeybindings()= helper and
   =keybindings:suggest= event, the merge logic, =defaults.json=, the
   =/kb bindings= overlay, and the =alt+space= / =alt+,= global
   shortcuts that work in insert-only mode.
2. *Modal editor* — a full =VimEditor= =CustomEditor= subclass
   (≈2000+ lines): grammar (operators, motions, text objects, counts,
   dot-repeat, visual / visual-line), Insert↔Normal switching,
   =alt+escape= abort handling, settings persistence, and the
   =setEditorComponent= wiring.

Both responsibilities share the parent task body. They also share a
single user settings file (=~/.pi/agent/keybindings-ext.json=), a single
slash command surface (=/kb …=), and a single suggestion API consumed by
every other extension. That coupling makes it harder to:

- Reason about which extension a user actually needs (most users never
  touch modal mode).
- Evaluate the upstream =kepatrick/vim-motions-pi= package as a
  drop-in replacement for our hand-rolled modal editor without first
  untangling it from the leader-menu code.
- Swap implementations in either half without a deep rebase across
  unrelated logic.

The parent TASKS.org task (=00e1772c=) records two threads — split,
and =vim-motions-pi= adoption. This change-record commits to a
*decision-then-split* sequence: the package evaluation runs first,
its outcome shapes how the modal half is extracted, then the leader
side is extracted in parallel. Breaking changes are acceptable in
exchange for clean naming and a one-step migration; in-tree consumers
are updated atomically with the split.

** Design decisions

- *Two new extensions, not three* :: =leader-menu= owns leader menus +
  cross-extension API; =vim-mode= owns the modal editor.  Shared
  helpers live in =agents/pi/extensions/lib/= as today.
- *Decision before split* :: =vim-motions-pi= is evaluated against the
  full feature set of our =VimEditor= before the modal extraction
  starts.  The evaluation outcome (ADOPT / EXTEND / KEEP_OWN) is
  recorded in =* Implementation= and selects the modal-extension
  body.  This keeps us from building =vim-mode= twice.
- *Breaking changes, one-step migration* :: slash commands, settings
  file, and event names may be renamed freely; every in-tree consumer
  is updated in the same change-record.  Migration notes live in
  =* Implementation= and the new extensions' README files.
- *Cross-extension API stability via the helper, not events* ::
  =suggestKeybindings()= in =lib/pi-utils.ts= keeps its current
  signature so consumers do not break.  Internally it forwards to
  =leader-menu=.  The =keybindings:suggest= / =keybindings:ready=
  event names may change; all in-tree emitters move atomically.
- *Final state: always-on when loaded* :: The initial split preserved
  a runtime vim-mode toggle for migration safety. Follow-up commit
  =3939073= deliberately removed it: =vim-mode= is now a
  single-responsibility extension that installs the modal editor when
  loaded; disabling means not loading the extension. The only migrated
  setting is =debug=, now owned by =leader-menu.json=.
- *Drop =/kb= entirely* :: New extensions register =/leader-menu= and
  =/vim-mode= directly; no alias shim, no deprecation notification.
  Documented in the migration pass.  (Decided 2026-05-02.)
- *=alt+escape= shim disposition deferred* :: Resolution depends on
  the evaluation outcome (task =16c3fbee=).  ADOPT → verify
  =vim-motions-pi='s abort handling and delete the shim;
  EXTEND/KEEP_OWN → the shim stays as-is.  (Decided 2026-05-02.)
- *=keybindings.org= replaced by an auto-export* :: Hand-maintained
  reference removed; =/leader-menu bindings= grows an export-to-org
  mode (e.g. =/leader-menu bindings --export=) that prints the table
  in the same format.  Eliminates drift between =defaults.json= +
  runtime suggestions and the doc.  (Decided 2026-05-02.)

* Implementation

** Outcome

The single =keybindings= extension (~3000 LOC monolith) was replaced
by two focused extensions:

- =leader-menu/= (~700 LOC) — leader-chord discovery, dispatch, the
  cross-extension =registerLeaderMenu()= API consumed by =tasks=,
  =term=, =git-diff=, and the new =/leader-menu bindings --export=
  org-format dump.
- =vim-mode/= (~2100 LOC) — the modal editor, with the existing
  =VimEditor= grammar preserved per the KEEP_OWN evaluation outcome
  and one upstream-inspired addition (repeat-find-backward =,=). It
  is always on when loaded and has no settings file or runtime toggle.

The two extensions are independently loadable and communicate purely
through events (=leader-menu:register=, =leader-menu:ready=,
=leader-menu:keys-resolved=, =leader-menu:open=). The hand-rolled
=extensions/keybindings.org= reference table is gone, replaced by
=/leader-menu bindings --export= which prints the live-merged tree.

Final settings shape:
- =~/.pi/agent/leader-menu.json= owns =globalLeader=, =localLeader=,
  and the shared key-dispatch =debug= flag.
- =~/.pi/agent/vim-mode.json= no longer exists. =leader-menu= migrates
  only =debug= from legacy =vim-mode.json= / =keybindings-ext.json=
  and intentionally drops =modal=.

Deliberate UX shift: bare configured leaders in Normal mode now open
leader-menu's standard centered overlay instead of the legacy
500 ms in-editor widget hint. Caveat from final review: the default
local leader =,= conflicts with Vim repeat-find-backward, so in Normal
mode comma is handled by Vim grammar first; use =alt+,= or configure
=localLeader= to a non-conflicting key (for example =;=) for bare
Normal-mode local menu access. Captured below as a follow-up candidate
if the default should change.

** Cross-extension delegation — =leader-menu:open= (added in task =0634cd5f=)

The split-contract task originally said leader-menu had two inbound
events (=leader-menu:register=) and one outbound (=vim-mode:enable=,
via the action dispatcher). During task =7c85f7fd= the design grew a
third inbound event so vim-mode could delegate bare-Space/=,= chords
in Normal mode without sharing leader-tree state:

#+begin_src typescript
  // leader-menu/index.ts
  pi.events.on("leader-menu:open", async (data: { rootKey?: string }) => {
    if (!currentCtx) return;
    await showLeaderOverlay(currentCtx, data?.rootKey ?? " ");
  });

  // vim-mode/index.ts (handleNormalMode)
  if (key === " " || key === ",") {
    this.emitLeaderOpen?.(key);
    return;
  }
#+end_src

This replaces the legacy =500 ms widget hint= UX inside =VimEditor=
with leader-menu's standard centered overlay (which is what
=alt+space= already opens in Insert mode). The two extensions stay
fully decoupled: no shared leader-tree state, no shared imports.
Follow-up captured below if a richer chord-dispatch protocol is
wanted later.

*** Cross-extension toggle contract (event-only, no imports)

The leader-menu's =Space t E v= / =Space t E e= chords drive a vim-mode
toggle without either extension importing the other. Mechanism uses
pi's existing event bus.

#+begin_src
  leader-menu/defaults.json
  │  Space t E v  → action: "vim-mode:enable"
  │  Space t E e  → action: "vim-mode:disable"
  ↓
  leader-menu/index.ts buildAction()
  │  action without command:/passthrough: prefix
  │  → pi.events.emit("vim-mode:enable", {})
  ↓
  vim-mode/index.ts pi.events.on("vim-mode:enable", …)
     → saveUserSettings({ modal: true })
     → ctx.ui.setEditorComponent(VimEditor factory)
     → ctx.ui.notify("Vim mode: on", "info")
#+end_src

Payload shape (locked in):

#+begin_src typescript
  // vim-mode:enable
  interface VimModeEnableEvent {
    /** Optional caller hint surfaced in the notification. */
    source?: "leader-menu" | "slash-command" | "settings-file" | "migration";
  }

  // vim-mode:disable
  interface VimModeDisableEvent {
    source?: "leader-menu" | "slash-command" | "settings-file" | "migration";
  }
#+end_src

The =source= field is purely advisory — it lets the notification say
"Vim mode enabled (from leader menu)" vs "Vim mode enabled (loaded
from settings)". =vim-mode= MUST tolerate an empty payload (the
leader-menu's =buildAction()= helper emits =emit(name, {})= with no
field). Both consumers and emitters are in-tree, so the payload can
be extended later without a deprecation cycle.

*** Settings migration

=vim-mode= ships a one-shot migrator that runs at
=session_start=:

1. If =~/.pi/agent/vim-mode.json= already exists → no-op.
2. Else if =~/.pi/agent/keybindings-ext.json= exists → read =modal=
   and =debug=, write =vim-mode.json= with the same values, delete
   the old file, =ctx.ui.notify()= once with a one-line migration
   note.
3. Else → do nothing; defaults apply (=modal: false=, =debug: false=).

=tasks/overlay.ts='s direct =debug= read updates to read
=vim-mode.json= in step with the rename. The migration ensures
freshly-cloned dotfiles work even when neither file exists yet.

*** Helper rename + verb alignment

=lib/pi-utils.ts::suggestKeybindings()= is renamed to
=registerLeaderMenu()= so the verb matches pi's existing extension
API (=registerCommand=, =registerTool=, =registerShortcut=). Old
name was a misnomer: the helper only ever handled leader-menu
entries, never arbitrary keybindings.

Other consequences of the rename:

- TypeScript type =KeybindingSuggestion= → =LeaderMenuRegistration=.
- Internal event =keybindings:suggest= → =leader-menu:register=
  (matches the helper verb).
- =keybindings:ready= → =leader-menu:ready= unchanged in role: it's
  a lifecycle signal, not the registration event itself. The helper
  subscribes to =:ready= and re-emits =:register= so reloads
  re-attach contributed menus.
- Argument shape and return value (=cleanup: () => void=) stay
  byte-identical so call sites need a one-line import + identifier
  swap, no signature edits.

In-tree call sites to update (3): =tasks/index.ts:38,769=,
=term/index.ts:11,855=, =git-diff/index.ts:14,268=. All use the

** Evaluation: =kepatrick/vim-motions-pi= vs current =VimEditor=

Date: 2026-05-02. Source examined at
[[https://github.com/kepatrick/vim-motions-pi][github.com/kepatrick/vim-motions-pi]]
@ v0.1.3 (commit fetched 2026-05-02). LOC: ~2400 across

*** Gaps that block straight ADOPT

Using =vim-motions-pi= as-is would lose, in order of importance to
me:

- *Dot-repeat (=.=)* — daily-use; the README explicitly lists this as
  unsupported.
- *Indent/dedent operators (=>= / =<=)* — useful in code editing.
- *Bracket / brace / quote text objects =i( i[ i{ i' i`=* — key for
  =ci(=, =da{= idioms.
- *=BIG-WORD= variants =W B E=* — frequently used for path-like
  tokens.
- *Visual case toggle =u U=* — occasional but absent.
- *=alt+escape= abort key* — the package uses bare =Esc= as the
  abort, but =Esc= is also the mode-switch key, so abort-from-normal
  has no path. We deliberately replaced bare =Esc= with =alt+escape=
  for this reason.

Wins from =vim-motions-pi= we don't currently have:

- *Clipboard sync* (=VIM_MOTION_PI_CLIPBOARD=, =/vim-clipboard=) —
  genuinely nice; would be cheap to port if we KEEP_OWN.
- *Repeat-find-backward =,=* — trivial gap to close on our side.
- *Smaller, decomposed code* (=core.ts=, =buffer.ts=, =clipboard.ts=)
  vs our 2000-line monolithic =VimEditor=. Better maintainability
  shape if we restructured.

*** Decision: KEEP_OWN

The gap list above represents about 30% of my own daily Vim usage
in pi (especially dot-repeat and bracket text objects). Adopting
=vim-motions-pi= as-is would be a net regression in my workflow;
adopting it as a base and re-adding the missing features in a
downstream wrapper (EXTEND) would mean maintaining a fork of a
moving target plus our own deltas — strictly more work than just
shipping our existing =VimEditor= as the new =vim-mode= extension.

*Decision*: =KEEP_OWN=. Move our =VimEditor= verbatim into the new
=vim-mode= extension (task =7c85f7fd=). Steal a couple of
=vim-motions-pi='s ideas while we're at it:

- Adopt its decomposition shape: split =vim-mode/index.ts= into
  =index.ts= (entry + editor class) + =buffer.ts= (text manipulation
  primitives) + =core.ts= (pure motion math) so future maintenance
  matches the upstream's cleaner factoring.
- Port =VIM_MOTION_PI_CLIPBOARD=-style clipboard sync as a follow-up
  =vim-mode= feature — not in scope for this refactor, captured as
  a new follow-up task below.
- Add =,= (repeat-find-backward) to our grammar; small win.

*Implications for other plan tasks*:

- The =alt+escape= shim *stays* (resolves the deferred design

- The =vim-motions-pi= adoption thread on the parent TASKS.org task

  bullet is satisfied with a documented "not adopted, here's why"

- Task =7c85f7fd= body switches to the KEEP_OWN branch.
