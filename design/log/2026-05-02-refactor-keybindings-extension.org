* Context

The =keybindings= pi extension under =agents/pi/extensions/keybindings/=
has grown into two distinct responsibilities living in one ~3000-line
=index.ts=:

1. *Leader-menu / which-key* — discoverable Space and =,= leader chords,
   the cross-extension =suggestKeybindings()= helper and
   =keybindings:suggest= event, the merge logic, =defaults.json=, the
   =/kb bindings= overlay, and the =alt+space= / =alt+,= global
   shortcuts that work in insert-only mode.
2. *Modal editor* — a full =VimEditor= =CustomEditor= subclass
   (≈2000+ lines): grammar (operators, motions, text objects, counts,
   dot-repeat, visual / visual-line), Insert↔Normal switching,
   =alt+escape= abort handling, settings persistence, and the
   =setEditorComponent= wiring.

Both responsibilities share the parent task body. They also share a
single user settings file (=~/.pi/agent/keybindings-ext.json=), a single
slash command surface (=/kb …=), and a single suggestion API consumed by
every other extension. That coupling makes it harder to:

- Reason about which extension a user actually needs (most users never
  touch modal mode).
- Evaluate the upstream =kepatrick/vim-motions-pi= package as a
  drop-in replacement for our hand-rolled modal editor without first
  untangling it from the leader-menu code.
- Swap implementations in either half without a deep rebase across
  unrelated logic.

The parent TASKS.org task (=00e1772c=) records two threads — split,
and =vim-motions-pi= adoption. This change-record commits to a
*decision-then-split* sequence: the package evaluation runs first,
its outcome shapes how the modal half is extracted, then the leader
side is extracted in parallel. Breaking changes are acceptable in
exchange for clean naming and a one-step migration; in-tree consumers
are updated atomically with the split.

** Design decisions

- *Two new extensions, not three* :: =which-key= owns leader menus +
  cross-extension API; =vim-mode= owns the modal editor.  Shared
  helpers live in =agents/pi/extensions/lib/= as today.
- *Decision before split* :: =vim-motions-pi= is evaluated against the
  full feature set of our =VimEditor= before the modal extraction
  starts.  The evaluation outcome (ADOPT / EXTEND / KEEP_OWN) is
  recorded in =* Implementation= and selects the modal-extension
  body.  This keeps us from building =vim-mode= twice.
- *Breaking changes, one-step migration* :: slash commands, settings
  file, and event names may be renamed freely; every in-tree consumer
  is updated in the same change-record.  Migration notes live in
  =* Implementation= and the new extensions' README files.
- *Cross-extension API stability via the helper, not events* ::
  =suggestKeybindings()= in =lib/pi-utils.ts= keeps its current
  signature so consumers do not break.  Internally it forwards to
  =which-key=.  The =keybindings:suggest= / =keybindings:ready=
  event names may change; all in-tree emitters move atomically.
- *Default-off modal* :: =vim-mode= remains opt-in.  Removing the
  =keybindings= extension does *not* enable modal editing for users
  who did not set =modal: true=.  Settings migration is best-effort
  with a one-shot warning if the legacy file is found.

human reference. =which-key= is the more natural home, but only if
