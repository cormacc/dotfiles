** Design decisions

- =CLOSED:= placement :: Standard org-mode places =CLOSED: [timestamp]=
  on its own line *between the heading and the =:PROPERTIES:= drawer*
  (and =org-todo= writes it there). The =org-tasks= skill currently
  mandates the opposite ("immediately after the =:END:= of the
  properties drawer"), which neither stock org nor the tasks extension's
  serializer follows. We align the skill (and any code that depends on
  the wrong wording) with org's native convention. The extension's
  existing serializer behaviour is correct and stays as-is.
- Creation timestamp :: New tasks record their creation time as a
  =:CREATED: [YYYY-MM-DD Day HH:MM]= property in the properties
  drawer rather than as inline prose at the top of the description.
  This keeps the heading's body free for human/agent notes and makes
  the timestamp machine-readable for filtering and sorting. The
  format matches =:STARTED:= and =CLOSED:= for consistency.
- Single source of truth for file format :: =org-tasks= SKILL is
  authoritative for file shape, properties, keywords, and the
  =#+SELECTED:= protocol. The tasks-extension README and =org-plan=
  SKILL link to =org-tasks= rather than restating the format.
- Single source of truth for the change-record skeleton :: One
  canonical skeleton (matching =org-plan='s "Minimal skeleton") is
  asserted by snapshot tests in both the TS scaffolder and the elisp
  scaffolder, so future drift becomes a CI failure rather than a docs
  inconsistency.
- Emacs and the extension stay independent at runtime :: No live IPC,
  no shared template files; both implementations read =TASKS.org=
  directly. Parity is enforced by tests against documented invariants,
  not by a runtime dependency.
- Terminology :: Skill and user-facing prose use *change-record*
  exclusively. Internal symbol names in code (=plan=,
  =tasks-org-plans-directory=, =scaffoldPlan=) are stable API and stay
  as-is to avoid a churny rename.
