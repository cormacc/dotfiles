* Summary
Treemacs-backed Emacs UI folded into the existing =tasks-org= Spacemacs package, giving feature parity with the pi tasks extension's expanded UI for navigate / expand / select / cycle workflows over the org-memory protocol. The UI lives in a dedicated =*tasks-org*= buffer, reads the graph from =TASKS.org= + =TASKS.local.org= + linked change-records, mutates through a small protocol helper surface, and refreshes via =file-notify= rather than RPC with pi. v1 also covers archive, publish/unpublish, doctor, task creation, details pane, linked-issue badges + =J= browser open, and a compact selected-task indicator.

** Decisions
- Fold UI into =tasks-org=, no sibling package :: one graph reader, one writer surface, one Nix slice when packaging arrives.
- Match pi UX semantics, not its rendering :: keys and behaviour mirror =pi/extensions/tasks/README.md=; rendering is Emacs-idiomatic.
- Files remain canonical; no Emacs↔pi RPC :: pi refreshes via its file watcher; Emacs refreshes via =file-notify=.
- Emacs writes co-canonical with pi for v1 :: whole-flip to Emacs-canonical mutation rejected; staged partial flip (status transitions only) is the v2 candidate if drift becomes observed.
- ECA interop is loose and file/cursor-mediated for v1 :: =e= → source heading is sufficient passive context; explicit ECA bridge deferred.

** Shipped
- =tasks-org= folds the new UI alongside the existing minor mode under =editors/emacs/spacemacs/layers/org-user/local/tasks-org/=.
- Treemacs renderer (Spacemacs binding =SPC a T= global, =, ; T= in =org-mode=); evilified buffer; excluded from =tasks-org-mode='s filename auto-enable.
- Transparent interop: launching from a =tasks-org-mode= buffer positions the cursor on the task at point; =e= jumps to source; =org-id-update-id-locations= runs over the loaded fileset on graph load.
- Parity ports: archive (=A=), publish/unpublish (=P=/=U=), doctor (=D=), task creation (=n=/=N=), details pane (=i=), linked-issue badges + =J= browser open, compact widget (=tasks-org-ui-show-selected=).
- Status cycling stays on the org-memory contract: protocol writer emits LOGBOOK / =CLOSED:= / =:STARTED:= identical to pi-side semantics.
- Spacemacs binding-collision fix: =h=/=l= cycle keys were shadowed by Treemacs tree-nav; promoted =S-<right>=/=S-<left>= as primaries (mirrors =org-shiftright=/=org-shiftleft=).

** Gotchas
- =vui.el= proved insufficient for hierarchy readability and was replaced by Treemacs mid-flight (plan task =c0459781=); graph reader, protocol helpers, =#+SELECTED:= / =#+IMPORT:= semantics, compact widget, and launch commands carried over unchanged.
- =org-entry-get= returns timestamps with surrounding brackets; archive code strips them before re-emitting =:ARCHIVED:= to keep single-bracket form. Caught by end-to-end ERT.
- The doctor's blocker reader was scanning forward for the next heading from POS at column 0 and matching the *current* heading — empty heading region, blockers never seen. Fixed by advancing one line before the next-heading scan.
- =split-string= clobbers global match data; capture =match-beginning= / =match-string= before calling it.
- A plan file without =#+TODO:= keeps =org-todo-keywords-1= at the org default; reader uses a protocol-keyword union so STARTED / WAITING / CANCELLED still surface.
- =tasks-org-ui.el= is =:no-byte-compile= because Treemacs' macros expand at load time and may not be available in arbitrary batch contexts. The Spacemacs =:after (treemacs tasks-org)= load order keeps runtime correct.
- The folded =tasks-org= still lives under the Spacemacs =org-user= layer; relocation captured as an open question.

- =OPEN= Should the folded =tasks-org= package relocate out of the Spacemacs =org-user= layer? (resolve before Nix packaging work).
- =OPEN= Should status cycling go through =org-todo= or stay on the custom writer?
