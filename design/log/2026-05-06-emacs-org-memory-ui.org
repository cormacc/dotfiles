* Context

The pi =tasks= extension (see =agents/pi/extensions/tasks/README.md=) is the canonical UI today over the plain-org task-memory protocol owned by the =org-tasks= / =org-plan= / =org-jira= skills. It provides a centred split-pane tree+details view, status cycling, single-selection via the checkout-local =#+SELECTED:= pointer in =TASKS.local.org=, change-record open/create via =#+IMPORT: [[file:...]]= links (=p=), archive (=A=), publish/unpublish (=P=/=U=), linked-issue badges + browser open (=J=), doctor checks, slash commands, leader-menu keybindings, and a compact selected-task widget.

Emacs already has a thin minor-mode layer at =editors/emacs/spacemacs/layers/org-user/local/tasks-org/tasks-org.el= that covers some file-level helpers: =:ID:= ensure, =#+SELECTED:= toggle, open existing =#+IMPORT:= change-records, publish/unpublish local tasks, and jump-to-parent. Those helpers piggy-back on ordinary org-mode editing and are useful prior art, but they are not a stable compatibility surface: selection writing is currently too destructive for local drafts, there is no Emacs-side plan scaffold helper, and status transitions only cover part of the org-memory lifecycle. Since there is little legacy use to preserve, this UI work may replace or substantially refactor the existing helpers where that produces a cleaner implementation.

This change-record plans a *visualisation/UI layer* added to the existing =tasks-org= package, rendered via Treemacs/treelib. The goal is feature-parity with the pi tasks extension's expanded UI for read/navigate/cycle workflows while delegating durable state mutation to a refactored =tasks-org= protocol helper surface and plain org editing where they fit.

** Design decisions

- Fold the UI into =tasks-org=, do not create a sibling package :: Considered keeping the new UI orthogonal as a separate package that depends on =tasks-org=, and rejected. The two pieces share the same domain (the org-memory protocol on the Emacs side) and should share one graph reader plus one set of protocol writers. A single package also mirrors the pi-side decision in the package-boundary plan (=2026-05-06-package-org-memory-protocol-extensions=) to bundle =tasks= with =emacsclient= rather than keep them split, and keeps the future Nix packaging slice clean (one Emacs deliverable next to the pi extensions). Concretely: the existing =tasks-org-mode= minor mode (file helpers in any org task buffer) and a new =tasks-org-ui-mode= derived major mode (dedicated UI buffer) cohabit under the existing namespace; both modules import a shared internal graph reader.
- Replace helper internals where needed, keep only a small protocol API :: Current =tasks-org= helpers are allowed to change or disappear if they are a poor fit. The stable boundary should become protocol-level functions used by both the minor mode and UI: load graph, locate task by UUID/source, read/write =#+SELECTED:= without dropping local drafts, open/create =#+IMPORT:= change-records, and transition status with org-memory LOGBOOK / CLOSED / =:STARTED:= semantics. The UI module must call those helpers rather than duplicating parser/writer code, but no legacy helper name is sacred.
- Match the pi tasks extension's UX semantics, not its rendering :: Key bindings, status cycling, selection behaviour, default collapse rules, and the split-pane shape should mirror =agents/pi/extensions/tasks/README.md= so muscle memory transfers. The rendering technology is Emacs-idiomatic (Treemacs / overlays / buttons), not a port of the pi widget tree.
- Render via Treemacs/treelib :: The expanded UI is built on Treemacs' extension API (=treemacs-define-entry-node-type=, =treemacs-define-expandable-node-type=, =treemacs-define-leaf-node-type=), which supports non-file hierarchical nodes with per-node labels, keys, properties, icons, and RET/TAB actions. Use a dedicated =*tasks-org*= buffer rather than embedding tasks in the normal Treemacs project sidebar, so file/workspace assumptions do not leak into the org-memory UI.
- Read-mostly v1 with a minimal mutation substrate :: First cut focuses on navigate / expand / select / cycle and plan/edit jumps. Unlike the existing helper set, v1 may add a minimal Emacs-side =#+IMPORT:= scaffold helper so =p= can create an empty change-record skeleton; agent-assisted plan drafting remains deferred. Archive (=A=), publish/unpublish (=P=/=U=), agent-assisted plan expansion, task creation (=n=/=N=), and the doctor command are deferred to follow-up tasks unless they fall out cheaply from the chosen rendering layer.
- Files remain canonical :: As with =tasks-org=, the UI saves through ordinary org file writes. Pi's file-watcher refreshes the pi widget; Emacs refreshes via =auto-revert= on the buffers the UI is reading. No Emacs↔pi RPC.
- Emacs writes are co-canonical with pi-side writes for v1, not primary :: Considered making =tasks-org= the canonical writer with the pi extension delegating mutations through =emacsclient= / =emacs_eval=, retaining pi-side writers as a non-Emacs-user fallback. Rejected for v1. Rationale:
  - The v1 UI is read-mostly. Mutation parity with the pi extension is itself a follow-up workstream. Inverting canonical ownership of operations the UI doesn't yet implement is premature.
  - Pi-side parser/writer drift is hypothetical, not observed; the pi extension has comprehensive parser/lifecycle/scaffold/doctor test suites. Flipping pre-emptively spends complexity on an unverified problem.
  - Even with a fallback, a flip *adds* an Emacs-delegation layer on top of pi's existing writers — it does not remove the pi writers, so the surface to keep aligned grows rather than shrinks.
  - A safer staged path exists: if drift or org-mode hook-ecosystem needs become real, flip *just status transitions* first (the strongest single case — =org-todo= owns LOGBOOK / CLOSED / =:STARTED:= exactly) and measure before expanding.
  Triggers that would justify revisiting (whole-flip or partial):
  - An observed parser-drift bug producing non-round-trip-stable files between pi-side writes and Emacs reads (or vice versa).
  - Demand for org-mode lifecycle integrations the pi-side writer cannot honour (custom =#+TODO:= sequences, =org-after-todo-state-change-hook= consumers, agenda inclusion rules, =org-attach= flows).
  - Latency tolerance test: the operations targeted for delegation must complete via =emacsclient= within the latency budget that keeps the pi UI feeling instant (rule of thumb: <100ms p95 round trip on a warm local Emacs server).
  v2 candidate scope: status-transition delegation only, recorded under follow-ups; whole-flip is not on the roadmap unless the partial flip proves the model.
- Spacemacs binding scheme :: Two complementary launchers, both registered through the standard Spacemacs declarations:
  - Global launcher =SPC a T= (applications → tasks) so the UI is reachable from any buffer without colliding with Spacemacs' existing =SPC a t= prefix. Uses =spacemacs/declare-prefix= + =spacemacs/set-leader-keys=.
  - In-context alias =, ; T= added to the existing =spacemacs/set-leader-keys-for-major-mode 'org-mode= block in =editors/emacs/spacemacs/layers/org-user/packages.el= alongside the current =;s=, =;p=, =;P=, =;t=, =;Lp=, =;Lu= bindings, so users inside =TASKS.org= or a change-record reach the UI from the same prefix as the rest of =tasks-org=.
- UI buffer is an evilified Spacemacs buffer :: The expanded UI should use a dedicated Treemacs/treelib-backed buffer with a task-specific mode/map, while preserving Spacemacs evilified motion semantics. The mode is registered with =spacemacs/evilify-map= (or the =evilified-state-evilify-map= helper) so vim motion keys remap to UI actions while =SPC= remains the global leader. Do NOT bind =SPC= as a local toggle — it would shadow the leader. The UI buffer must also be excluded from =tasks-org-mode='s filename auto-enable so the minor mode does not run inside the visualisation buffer.
- Transparent interop with org source buffers :: Crossing between source org buffers and the UI must preserve location and stay live without a manual refresh:
  - Launching the UI from a =tasks-org-mode= buffer positions the cursor on the task at point.
  - The UI re-renders automatically when =TASKS.org=, =TASKS.local.org=, or any loaded change-record changes on disk (via =file-notify= watchers), preserving the cursor task by UUID across re-renders.
  - =e= jumps to the source heading using =pop-to-buffer-same-window= semantics that reuse an already-visible source window rather than splitting again.
  - Graph load proactively calls =org-id-update-id-locations= over the loaded fileset so UUID jumps work without the user having opened each file manually.
- Project root via projectile :: Same convention as =tasks-org=: =projectile-project-root= with a directory fallback. The UI loads the graph rooted at =TASKS.org= + linked change-records + =TASKS.local.org=.
- ECA interop is loose and file/cursor-mediated for v1 :: ECA already consumes editor file path + cursor position as implicit context, so pressing =e= in the UI to land on a task heading in its source buffer is sufficient passive interop — no UI ↔ ECA RPC, no graph serialisation. An explicit ECA bridge (e.g. a =c= key that opens a chat scoped to the cursor task's UUID, summary, change-record path, and =:LINKED_ISSUES:=) is a v2 candidate, recorded under follow-ups. Avoid duplicating the pi-side =tasks_insert_task= cross-extension primitive; if mutation-via-agent becomes desirable later, route it through the same files rather than a parallel API.
