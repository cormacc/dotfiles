* Context
This plan adds Emacs-side tooling for the plain-org task memory protocol used by
`TASKS.org`, linked plan files, and the pi tasks extension. The first
implementation should live inside the existing Spacemacs `org-user` layer at
`editors/emacs/spacemacs/layers/org-user/`, probably as a local package such as
`local/tasks-org/tasks-org.el`, with layer wiring in `packages.el` and
user-facing documentation in the layer README.

The goal is not to replace org-mode or make Emacs depend on pi. Editing should
continue to feel like ordinary org editing: users write normal headings,
properties, tags, and links. The Emacs package should provide transparent,
explicit helpers for the task-memory protocol: stable `:ID:` properties,
single-selection via the reserved `:selected:` tag, convenient `:PLAN:`
open/create workflows using clickable org links, and conservative save-time
support where it is safe. Pi should continue to refresh from disk through its
existing file watchers rather than requiring a new live protocol initially.

** Design decisions

- Lightweight minor mode, not a heavyweight replacement :: Implement a small
  `tasks-org-mode` minor mode so commands and keybindings are discoverable while
  leaving normal org-mode behavior intact.
- Keep implementation local for now :: Add the package to the existing
  `org-user` Spacemacs layer; design it so it can be extracted later with
  minimal coupling to the dotfiles repository.
- Prefer explicit commands over surprising hooks :: Automatic mutation should be
  limited and opt-in/conservative. Commands may update `:ID:`, `:selected:`, and
  `:PLAN:` explicitly; save hooks should only enforce safe invariants if enabled.
- File protocol remains canonical :: Emacs writes plain org files. Pi, agents,
  and humans consume the same files without requiring Emacs-specific state.
- Sync through files first :: The first version should rely on save-to-disk plus
  pi's existing file watchers. Any direct Emacs↔pi notification mechanism is a
  later enhancement, not a prerequisite.
- Spacemacs key prefix :: Bind minor-mode commands under the org local-leader
  prefix `, ;` in the `org-user` layer. Originally `, p` (blocked by
  `org-priority`), then `, K` (required ugly unbinding of `org-shiftup`).
  `, ;` is free, requires no unbinding, and is reachable without Shift.
  See the keybinding rework task (=b1c8e7a3=) for the full evaluation.
- Activation by filename match and keyword :: `tasks-org-mode` auto-enables
  on buffers matching `tasks-org-auto-enable-paths` (project-relative;
  defaults: `TASKS.org`, `design/log/*.org`) OR containing a
  `#+DEFAULT-PLAN-DIR:` keyword. The keyword check fires first and covers
  task-memory root files opened from outside their project root. Manual
  `M-x tasks-org-mode` remains available for other buffers.
- Project root via projectile :: Use `projectile-project-root` to locate the
  project root (and therefore `TASKS.org` and the plans directory) from any
  buffer. Fall back to the buffer's directory only when projectile cannot
  resolve a root.
- Subtask absorption deferred :: First-pass plan creation scaffolds an empty
  plan file only. Moving local subtasks into the new plan (preserving IDs and
  leaving a parent summary) is non-trivial and tracked as a separate
  workstream rather than v1 scope.
- ID backfill is not an Emacs responsibility :: The pi tasks extension
  backfills missing =:ID:= properties on load; the agent org-memory skill
  does the same on resume. Duplicating that in Emacs (as public commands or a
  save hook) adds noise without value for the common workflow. The only
  Emacs-side ID assignment is the internal call in =tasks-org--create-plan-for-current-task=
  which must stamp a new ID before pi sees the file for the first time.
  Public =tasks-org-ensure-id= / =tasks-org-backfill-ids-in-buffer= commands
  and the =tasks-org-save-backfill-ids= customisation were removed
  post-validation (2026-04-27).

Deferred follow-up. The current bindings (=, K {s,i,I,p,t}=) work but the
implementation is ugly:
- Capital =K= is not mnemonic for "tasks".
- =, p= (the original choice) was already bound by Spacemacs's org layer to
  =org-priority=; capital =, K= turned out to be bound to
  =org-kill-note-or-show-branches=.
- We unbind =, K= and then re-declare it as our prefix, deferring the whole
  sequence via =with-eval-after-load 'org= to win the timing race against
  =+lang/org='s late binding.

Acceptance criteria:
- A keybinding scheme that does not require unbinding existing Spacemacs
  org-layer bindings, OR
- A clearly justified scheme under a stable prefix that is unlikely to be
  reclaimed by upstream Spacemacs.
- README and plan design-decisions updated to match the new scheme.

Candidate approaches to evaluate:
- Move to a global leader (e.g. =SPC a t= for "applications > tasks") so
  bindings don't compete with org-mode prefixes at all.
- Use a special-character prefix (e.g. =, != or =, ;=) that Spacemacs does
  not bind in any layer.
- Nest under a sub-prefix already declared by Spacemacs (requires surveying
  which prefixes are designated for layer extension).
- Accept the current scheme as intentional and document the trade-off.

[2026-04-27 Sun] Adopted =, ;= as the prefix. =;= is free in the Spacemacs
org layer, reachable without Shift, and requires no unbinding of existing
bindings. The =with-eval-after-load 'org= timing workaround was removed
entirely. Candidate approaches evaluated and rejected: =SPC a t= (global
leader, too many keystrokes for frequent use), single free capital letters
(non-mnemonic). =;= is non-mnemonic too but is the most ergonomic free key.

Not blocking v1. Revisit when the package is stable enough that a breaking
key change is acceptable, or before sharing the package beyond personal use.

* Implementation

Implementation notes should be added as tasks are completed. Keep this section

v1 implementation summary (2026-04-25):

- Package lives at
  =editors/emacs/spacemacs/layers/org-user/local/tasks-org/tasks-org.el= and is
  wired into =org-user-packages= as =(tasks-org :location local)= with an
  =org-user/init-tasks-org= function that registers Spacemacs local-leader
  bindings under =, p=.
- Activation: =add-hook 'org-mode-hook #'tasks-org-maybe-enable= runs the
  filename-match check; matched buffers get =tasks-org-mode= enabled. The
  match list is a customisation (=tasks-org-auto-enable-paths=) defaulting to
  =TASKS.org= and =design/log/*.org= relative to the project root.
- Project root: =projectile-project-root= via =fboundp= guard so the package
  doesn't hard-require projectile; falls back to the buffer's directory and
  finally =default-directory=.
- ID helpers use =org-id-new= for UUID generation. =tasks-org-ensure-id= and
  =tasks-org-backfill-ids-in-buffer= rely on =org-entry-get/put=, leaving
  unrelated property drawers untouched.
- Selection: =tasks-org--clear-selected-everywhere= reads :PLAN: links from
  TASKS.org via a regex over property lines, then opens each file with
  =find-file-noselect= (killing buffers it opened itself), uses
  =org-set-tags= to mutate, and saves. =tasks-org-toggle-selected= clears
  before applying so single-selection is always enforced.
- Plan helpers: =tasks-org--extract-plan-path= recognises bare paths,
  =[[file:...]]=, and =[[file:...][label]]= forms; =::search-option= suffixes
  are stripped before resolving the file path (post-v1 fix, 2026-04-27).
  Plan creation always writes the clickable =[[file:...]]= form via
  =org-entry-put= and ensures the source task has an :ID: before saving.
  Subtask absorption deliberately not implemented.
- Save hook: =tasks-org--maybe-backfill-on-save= is registered buffer-locally
  by =tasks-org-mode= but only acts when =tasks-org-save-backfill-ids= is
  non-nil (off by default).

* Open questions (resolved 2026-04-27)

All resolved after manual end-to-end validation (=437d6716=).

- Save-time ID backfill scope :: *Removed entirely (2026-04-27).* The pi
  tasks extension and agent org-memory skill already backfill IDs
  automatically; duplicating the responsibility in Emacs is redundant.
  =tasks-org-ensure-id=, =tasks-org-backfill-ids-in-buffer=, the save hook,
  and the =tasks-org-save-backfill-ids= customisation were all removed.
  See the Context design-decisions section for rationale.
- Plan link parser scope :: *Strip search options (option b).* Added a
  =replace-regexp-in-string= call in =tasks-org--extract-plan-path= to drop
  =::...= suffixes before resolving the file path. Handles all realistic
  =:PLAN:= values without pulling in =org-element= for link parsing.
  Remaining edge case (label text containing =]=) accepted as negligible.
- Activation pattern coverage :: *Keyword-gated variant of option (c).*
  Added =tasks-org--has-plan-dir-keyword-p= which fires before the
  project-relative path check. =#+DEFAULT-PLAN-DIR:= is specific to the
  task-memory protocol so the trigger is tight with no false positives.
  Files outside the project that lack the keyword can still use
  =M-x tasks-org-mode= manually.
- Buffer cleanup correctness :: *Validated.* Interactive testing confirmed
  that =tasks-org--clear-selected-everywhere= correctly kills only the buffers
  it opened itself and leaves pre-existing visiting buffers intact. No stale
  modified-buffer issue observed.
- Subtask absorption follow-up :: *Tracked as separate workstream.* A TODO
  task has been added to =TASKS.org= for future implementation, to be
  coordinated with the pi tasks extension's parser.
