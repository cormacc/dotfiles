* Context
This plan adds Emacs-side tooling for the plain-org task memory protocol used by
`TASKS.org`, linked plan files, and the pi tasks extension. The first
implementation should live inside the existing Spacemacs `org-user` layer at
`editors/emacs/spacemacs/layers/org-user/`, probably as a local package such as
`local/tasks-org/tasks-org.el`, with layer wiring in `packages.el` and
user-facing documentation in the layer README.

The goal is not to replace org-mode or make Emacs depend on pi. Editing should
continue to feel like ordinary org editing: users write normal headings,
properties, tags, and links. The Emacs package should provide transparent,
explicit helpers for the task-memory protocol: stable `:ID:` properties,
single-selection via the reserved `:selected:` tag, convenient `:PLAN:`
open/create workflows using clickable org links, and conservative save-time
support where it is safe. Pi should continue to refresh from disk through its
existing file watchers rather than requiring a new live protocol initially.

** Design decisions

- Lightweight minor mode, not a heavyweight replacement :: Implement a small
  `tasks-org-mode` minor mode so commands and keybindings are discoverable while
  leaving normal org-mode behavior intact.
- Keep implementation local for now :: Add the package to the existing
  `org-user` Spacemacs layer; design it so it can be extracted later with
  minimal coupling to the dotfiles repository.
- Prefer explicit commands over surprising hooks :: Automatic mutation should be
  limited and opt-in/conservative. Commands may update `:ID:`, `:selected:`, and
  `:PLAN:` explicitly; save hooks should only enforce safe invariants if enabled.
- File protocol remains canonical :: Emacs writes plain org files. Pi, agents,
  and humans consume the same files without requiring Emacs-specific state.
- Sync through files first :: The first version should rely on save-to-disk plus
  pi's existing file watchers. Any direct Emacs↔pi notification mechanism is a
  later enhancement, not a prerequisite.
- Spacemacs key prefix :: Bind minor-mode commands under the org local-leader
  prefix `, p` (local-leader `p`) in the `org-user` layer.
- Activation by filename match :: `tasks-org-mode` auto-enables on buffers
  visiting `TASKS.org` or any `*.org` under a project's `design/log/`
  directory. Manual `M-x tasks-org-mode` remains available for other buffers.
- Project root via projectile :: Use `projectile-project-root` to locate the
  project root (and therefore `TASKS.org` and the plans directory) from any
  buffer. Fall back to the buffer's directory only when projectile cannot
  resolve a root.
- Subtask absorption deferred :: First-pass plan creation scaffolds an empty
  plan file only. Moving local subtasks into the new plan (preserving IDs and
  leaving a parent summary) is non-trivial and tracked as a separate
  workstream rather than v1 scope.

* Implementation

Implementation notes should be added as tasks are completed. Keep this section
focused on decisions, tricky Emacs/org API details, and validation outcomes that
future agents or humans need when maintaining the package.

v1 implementation summary (2026-04-25):

- Package lives at
  =editors/emacs/spacemacs/layers/org-user/local/tasks-org/tasks-org.el= and is
  wired into =org-user-packages= as =(tasks-org :location local)= with an
  =org-user/init-tasks-org= function that registers Spacemacs local-leader
  bindings under =, p=.
- Activation: =add-hook 'org-mode-hook #'tasks-org-maybe-enable= runs the
  filename-match check; matched buffers get =tasks-org-mode= enabled. The
  match list is a customisation (=tasks-org-auto-enable-paths=) defaulting to
  =TASKS.org= and =design/log/*.org= relative to the project root.
- Project root: =projectile-project-root= via =fboundp= guard so the package
  doesn't hard-require projectile; falls back to the buffer's directory and
  finally =default-directory=.
- ID helpers use =org-id-new= for UUID generation. =tasks-org-ensure-id= and
  =tasks-org-backfill-ids-in-buffer= rely on =org-entry-get/put=, leaving
  unrelated property drawers untouched.
- Selection: =tasks-org--clear-selected-everywhere= reads :PLAN: links from
  TASKS.org via a regex over property lines, then opens each file with
  =find-file-noselect= (killing buffers it opened itself), uses
  =org-set-tags= to mutate, and saves. =tasks-org-toggle-selected= clears
  before applying so single-selection is always enforced.
- Plan helpers: =tasks-org--extract-plan-path= recognises bare paths,
  =[[file:...]]=, and =[[file:...][label]]= forms. Plan creation always
  writes the clickable =[[file:...]]= form via =org-entry-put= and ensures
  the source task has an :ID: before saving. Subtask absorption deliberately
  not implemented.
- Save hook: =tasks-org--maybe-backfill-on-save= is registered buffer-locally
  by =tasks-org-mode= but only acts when =tasks-org-save-backfill-ids= is
  non-nil (off by default).

Key-binding map (Spacemacs local-leader =, p=):

| Key   | Command                            |
|-------+------------------------------------|
| , p s | tasks-org-toggle-selected          |
| , p i | tasks-org-ensure-id                |
| , p I | tasks-org-backfill-ids-in-buffer   |
| , p p | tasks-org-open-plan                |
| , p t | tasks-org-jump-to-tasks            |

* Open questions

Batch-review items collected during v1 implementation. Resolve before/after
interactive validation (=437d6716=).

- Save-time ID backfill scope :: Currently exposed as a customisation
  (=tasks-org-save-backfill-ids=) defaulting to =nil=. Decide whether to
  enable it by default for buffers that match =tasks-org-auto-enable-paths=,
  enable globally, leave opt-in, or remove the save hook entirely in favour
  of explicit commands.
- Plan link parser scope :: =tasks-org--extract-plan-path= uses a regex that
  captures everything between =file:= and the closing =]=. Org link forms
  with search options (=[[file:foo.org::heading]]=) will have the =::heading=
  included in the path and fail to open. Decide whether to (a) accept this
  v1 limitation, (b) strip search options before resolving the file, or
  (c) switch to =org-link-open-from-string= / =org-element-link-parser= for a
  fully org-aware parse.
- Activation pattern coverage :: =tasks-org-auto-enable-paths= matches paths
  relative to the project root. Files outside the project (e.g. a plan file
  visited via an absolute path from outside its repo) won't auto-enable.
  Decide whether to (a) accept the project-only limitation, (b) also try a
  match against =buffer-file-name= directly, or (c) gate activation on the
  presence of a =#+PLANS:= or =#+TODO:= keyword in the buffer.
- Buffer cleanup correctness :: =tasks-org--clear-selected-everywhere= kills
  buffers it opened itself but keeps pre-existing visiting buffers. Verify
  during interactive testing that this does not leave the user with stale
  modified buffers if a save fails partway.
- Subtask absorption follow-up :: Deferred per design decisions. When ready,
  open as a separate workstream with its own plan; coordinate with the pi
  tasks extension's parser so behaviour matches across editor and CLI.
