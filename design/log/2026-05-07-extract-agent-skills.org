* Context
The dotfiles repo currently carries rapidly changing agent assets under
~agents/~: generic skills, pi-specific skills, pi extensions, shared extension
helpers, prompt templates, package metadata, and user-local pi settings. Most
recent work is in that agent surface, while the rest of the dotfiles and system
configuration should remain relatively static.

This change extracts the reusable agent assets into a new dedicated repository
and flake, expected to be ~github:cormacc/dotagents~. The extracted repo should
ship the reusable source and packaging surfaces; this dotfiles repo should keep
only the user's local installation/configuration wiring and the pi
~settings.json~ file.

Scope includes:
- generic agent skills under ~agents/skills~;
- pi-specific skills under ~agents/pi/skills~;
- pi extensions under ~agents/pi/extensions~ plus shared helper modules such as
  ~agents/pi/extensions/lib~;
- prompt templates under ~agents/prompts~;
- package, README, Nix packaging, validation, and any manifest files needed for
  pi package installation and Vercel Labs ~skills~ installer compatibility.

Scope excludes:
- ~agents/pi/settings.json~, because it is user-local pi configuration;
- forcing the existing ~agents.mode = editable|packaged~ abstraction to survive
  in dotfiles. Dotfiles should always use editable symlink wiring after the
  migration.

The migration should preserve the existing runtime destinations:
- =~/.agents/skills= for generic cross-agent skills;
- =~/.pi/agent/extensions= for pi extensions;
- =~/.pi/agent/skills= for pi-specific skills;
- =~/.pi/agent/prompts= for prompt templates;
- =~/.pi/agent/AGENTS.md= if the extracted repo continues to provide pi-local
  agent instructions;
- =~/.pi/agent/settings.json= from this dotfiles repo.

** Design decisions
- Dedicated repository :: Use a new repo rather than a subdirectory-only flake so
  rapid agent iteration can move independently of stable system/dotfiles work.
- Editable install is the primary consumer contract :: The dotfiles integration
  must point symlinks at a live checkout of the new repo so skill and extension
  edits are reloadable in place without a Home Manager switch.
- Submodule + flake outputs :: Dotfiles consumes dotagents as a git submodule
  for the editable source checkout, while dotagents still exposes flake outputs
  for packages and checks. ~mkOutOfStoreSymlink~ targets the live submodule
  working tree rather than an immutable Nix store path.
- Disabled extensions stay in source, out of packaging :: Move
  ~agents/pi/extensions/disabled/~ as-is into the new repo, but exclude its
  contents from all exposing package scripts, ~package.json~ entries, the Nix
  package fileset, and any flake outputs.
- User-local settings stay local :: Do not move ~agents/pi/settings.json~ into
  the extracted repo; keep package/source registration and secrets/local toggles
  under the user's dotfiles checkout.
- Preserve public installer compatibility :: Keep generic skills discoverable by
  the Vercel Labs ~skills~ CLI. Its discovery supports root ~skills/~ directories,
  direct ~SKILL.md~ subpaths, and optional plugin manifests, so the extracted
  repo should keep a conventional ~skills/<name>/SKILL.md~ layout and document
  ~npx skills add cormacc/dotagents --list~ / selected-skill installs.
- Preserve pi package compatibility :: Continue exposing pi package metadata for
  pi extensions and skills, including any helper directories required for
  relative imports. Treat manifest/package membership as the source of truth.
  ~leader-menu~ has been retired from the active set into
  ~pi/extensions/disabled/leader-menu/~ and is excluded from the org-memory
  bundle accordingly.
- Keep Linux and Darwin first-class :: Flake outputs, Home Manager integration,
  package builds, and checks should cover both ~x86_64-linux~ and
  ~aarch64-darwin~.
