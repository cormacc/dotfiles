* Context

The current =/jira clone= path is a pure prompt-builder that asks the
agent to call =atlassian_getJiraIssue=, parse the response, render the
org task body, and emit an =edit= tool call to insert it into a
=TASKS*.org= file.  Token cost is dominated by:

1. /Ingestion/ of the MCP response (~17 KB of JSON for WEB-234, of which
   ~1.5 KB is consumed). Step 1 of this initiative — narrowing the
   =fields= list in the prompt — shipped at commit =2f0f354= on
   2026-04-28 and dropped this to ~1.5 KB.
2. /Emission/ of the rendered org body in the =edit= tool's =newText=
   argument.  For body-heavy issues this is the larger cost (~300–800
   tokens per clone before any cleanup).
3. /Re-derivation/ of deterministic transforms (priority mapping, UUID,
   ~CREATED~ timestamp, drawer assembly, label-tag synthesis) by the
   model on every clone.

The remaining wins from (2) and (3) are addressed here.

** Architectural reality (2026-04-28)

A registered pi tool's executor /cannot/ invoke MCP from JavaScript
today.  =pi-mcp-adapter='s MCP client (=connection.client.callTool=) is
internal to the adapter; the pi =ExtensionAPI= exposes no
=callTool= / =mcp.call= / cross-extension RPC primitive.  Any MCP fetch
must therefore remain brokered by the LLM.

Consequences for this plan:

- The fetch-side win envisioned by the original parent-task body
  (=jira_clone_fetch= as an extension-side wrapper that caches the
  filtered MCP response) is /not implementable/ with today's
  ExtensionAPI.  It is dropped from this plan.
- The emission-side win /is/ implementable: a registered
  =jira_clone_apply= tool with a structured-args schema can perform
  the deterministic transforms and the file write inside the
  extension, so the model never re-emits the rendered body.
- The "shared =tasks.insertTask= API" subtask is the dependency root
  for =jira_clone_apply= and is implemented first.
- ~/jira get~ (originally framed as a slash command paired with a
  =jira_get_issue= tool) is reduced to a prompt-builder slash command
  with no underlying tool.  It still validates keys, resolves cloudId,
  reuses the field-list constant, and asks the agent to render a
  compact human-readable summary.

If pi-mcp-adapter later exposes a JS-callable client to extensions, a
follow-up change-record can revisit the fetch-side win.  This plan
does not block on that.

** Design decisions

- /Re-scope: emission-side win only./  Decision (2026-04-28): drop
  =jira_clone_fetch= and any cache.  Implement =jira_clone_apply= +
  =tasks.insertTask= today; defer fetch-side optimisation until the
  ExtensionAPI gains a JS-callable MCP primitive.

- /=tasks.insertTask= owns deterministic org-task assembly./  Owned by
  the =tasks= extension, consumed by =jira_clone_apply= and any future
  cross-tracker integration (github/linear/gitlab).  Inputs are
  structured fields (=summary=, =priorityName=, =body=, =labels=,
  =linkedIssues=, =file=, =section=).  Performs priority mapping
  (Highest..Lowest → #A..#D), UUID generation, ~CREATED~ timestamp,
  drawer assembly, section-existence checks, idempotency on
  pre-existing =:LINKED_ISSUES:= match.

- /=jira_clone_apply= is the only jira-side runtime addition./  It
  takes the parsed Jira fields, performs Jira-specific transforms
  (priority-name → cookie via the existing table; label list → org
  tags), and delegates the org write to =tasks.insertTask=.  No
  org-mode string assembly inside the jira extension.

- /Existing =/jira clone= prompt becomes a two-step dispatch./
  =buildClonePrompt= is updated to instruct the agent to call
  =atlassian_getJiraIssue= (with the field filter shipped at
  =2f0f354=), then call =jira_clone_apply= with the parsed fields.
  No change to the user-facing slash command.

- /Idempotency./  =tasks.insertTask= refuses (with a structured error)
  when any task across =TASKS.org= or its imports already carries the
  same =:LINKED_ISSUES:= token.  The caller (=jira_clone_apply= or the
  user-facing flow) decides whether to surface this as a notification
  or transform into an update.

- /Body normalisation stays out of scope./  The model still owns the
  ADF→markdown cleanup heuristic surfaced in today's WEB-234 clone
  (broken =| --- |= rows, summary-trim).  =tasks.insertTask= writes
  whatever body it's given verbatim.  If the cleanup heuristic
  becomes deterministic enough later, it can move into the helper.

- /Sandbox-only validation./  Smoke tests run against project =SAND=
  only, per the org-jira skill.

* Plan

Top-level plan tasks below mirror the parent task's TASKS.org subtask
=:ID:= values one-to-one (per the org-plan / org-tasks UUID-mirroring
rule).  Finer-grained subtasks beneath them get fresh UUIDs and
represent plan-only work units.

* Implementation

** Architecture

- /Pure helper at the bottom./ =agents/pi/extensions/tasks/insert.ts=
  exposes =buildTaskBlock= (org-mode rendering, no I/O) and
  =insertTaskIntoFile= (file-side splice + idempotency). Both are
  importable JS — no =pi-tui= / =pi-coding-agent= dependency — so
  cross-extension consumers can use them without round-tripping
  through the LLM tool registry.

- /LLM-tool wrappers on top./ =tasks_insert_task= (in =tasks/index.ts=)
  and =jira_clone_apply= (in =jira/index.ts=) are thin TypeBox-schema
  adapters around those JS helpers. =jira_clone_apply= calls
  =insertTaskIntoFile= directly via JS import; the duplication of
  the schema for the LLM is intentional, since the contract the
  model sees needs Jira-specific descriptions (e.g. priority-name
  buckets) that are out of scope for the generic helper.

- /Two-step dispatch in the prompt./ =buildClonePrompt= no longer
  asks the agent to render an org task block via =edit=. Instead it
  describes the structured =jira_clone_apply= args and the
  =inserted=/=duplicate=/=section_not_found=/=error= return shapes.
  This is the emission-side win: the rendered org body never
  appears in any =edit= tool call.

** Idempotency contract

=insertTaskIntoFile= scans the target file plus every =alsoScan=
entry (and recursively their =#+IMPORT:= references) for any task
whose =:LINKED_ISSUES:= drawer property contains any of the
requested tokens. First match wins; the helper refuses with
=status: "duplicate"= and returns =existingId= + =existingFile= +
=conflictingToken=. The file is not mutated on refusal.

=jira_clone_apply= passes =alsoScan: [TASKS.local.org]= when the
target is =TASKS.org= (and vice-versa) so a draft of the same issue
in either slot trips the refusal regardless of which file the user
cloned to first.

** Token-reduction reasoning

Qualitative break-down of model emission per clone:

| Step                                   | Pre-2f0f354    | Post-2f0f354 (today)        | Post-this-change          |
|----------------------------------------+----------------+-----------------------------+---------------------------|
| Issue ingest (atlassian_getJiraIssue)  | ~17 KB JSON    | ~1.5 KB JSON (field filter) | ~1.5 KB JSON (unchanged)  |
| Org body emission                      | edit tool: drawer + heading + UUID + timestamp + body (~300–800 tok) | edit tool: same as pre (~300–800 tok) | tool args: key + summary + priorityName + body + labels (body is the only large field; org boilerplate gone) |
| Idempotency on re-clone                | none           | none                        | structured refusal at the helper, no model rendering at all |

The single largest residual cost is the issue body itself, which is
intrinsic. The savings come from removing the org-mode boilerplate
around it (=:PROPERTIES:= drawer, UUID, =:CREATED:= timestamp,
=:LINKED_ISSUES:=, heading priority-cookie syntax) plus the ability
to refuse duplicates without invoking the model for rendering.

An empirical measurement against a body-heavy SAND issue is captured
as the first =* Open question= above; once it lands the table can
be turned into concrete numbers.
