* Context

This draft plan was extracted from the org-memory agent-memory review
(=design/log/2026-05-01-org-memory-agent-memory-review.org=, parent task
=724c0332=). The generic protocol now has clear identity, LOGBOOK,
sandboxing, and resume semantics, plus regression coverage. One remaining
compatibility question is how strictly tooling should treat historical
org-task data that predates the current metadata rules.

The current protocol says new tasks require:

- =:ID:= UUID property.
- bracketed =:CREATED: [YYYY-MM-DD Day HH:MM]= property.
- tool-driven lifecycle history in =:LOGBOOK:= where possible.

Existing data may be older and incomplete. The tasks extension also has a
historical =backfillMissingIds()= path, which silently repairs missing IDs on
load. That may conflict with the newer “tolerate legacy reads, avoid silent
rewrites” posture and should be reviewed deliberately rather than folded into
the completed org-memory review.

** Design decisions to consider

- Separate read tolerance from write requirements :: New writes should stay
  strict and deterministic. Existing files should remain loadable even when
  metadata is incomplete.
- Avoid surprise file churn :: Any normalization that rewrites historical data
  should be explicit and user-initiated unless there is a strong compatibility
  reason to keep automatic ID backfill.
- Prefer diagnostics first :: =/tasks doctor= is the natural warning surface
  for malformed legacy metadata; users can decide whether to normalize.

* Open questions

** OPEN Should automatic missing-ID backfill remain enabled?
The extension currently has =backfillMissingIds()=. Decide whether this remains
a compatibility repair, moves behind an opt-in normalization command, or is
removed in favour of doctor diagnostics.

** OPEN What provenance should normalized =:CREATED:= timestamps carry?
Options include current time with a note, a git-history-derived lower bound, or
no automatic timestamp backfill at all.
