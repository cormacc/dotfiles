#+TODO: TODO(t) STARTED(s) WAITING(w) | DONE(d) CANCELLED(c)

* Context

Gap analysis of the org-memory and plan skills identified missing conventions,
ambiguous protocols, and unnecessary coupling between skill content and
extension-specific UI details.

A pre-implementation review also found a serious round-trip preservation concern:
when task files are saved by tooling, org metadata and non-task structure must not
be discarded. This plan therefore includes both protocol-documentation updates and
a tasks-extension verification/fix item.

** Design decisions

- *:selected: stays an org tag.* It is already established and managed by the
  tasks extension/human task-selection workflow. Agents should preserve and
  respect it, and should not create, move, or clear it unless explicitly asked or
  acting through a task-selection tool.

- *Blocking context uses :BLOCKED_BY:, not a :blocked: tag.* Tags are semantic
  category labels. The WAITING TODO keyword conveys blocked state; :BLOCKED_BY:
  records the reason or dependency in a machine-readable property.

- *Tasks carry :ID: UUID properties.* Use the standard org-mode :ID: property
  with UUID v4 values, compatible with org-id.el. IDs provide stable machine
  identity across renamed or restructured tasks.

- *Skills define protocol; extension implements UI.* Skill files should specify
  the file format contract and agent behaviour. Extension-specific UI details,
  keybindings, rendering, and safe write mechanics belong in the tasks extension.

- *Task memory files declare TODO keywords explicitly.* `TASKS.org` and newly
  scaffolded linked plan files should include `#+TODO: TODO(t) STARTED(s)
  WAITING(w) | DONE(d) CANCELLED(c)` so org-mode users get the same state cycle
  without relying on global Emacs configuration.

- *Plan files may include a `** Design decisions` subsection under `* Context`.*
  Use it when alternatives, constraints, or rationale will matter to future
  humans/agents. Keep it optional for trivial plans.

Extension implementation notes:

- Parser now tracks each parsed task subtree's original line range and source
  content.
- `serializeTasksPreservingFile` replaces existing root task subtrees in-place,
  removes omitted root tasks for archiving, and appends new root tasks while
  keeping non-task org content intact.
- Overlay saves, plan attachment, task creation, and archiving now use preserved
  serialization for TASKS.org/plan files.
- Extension-created tasks now include an :ID: UUID property.

UUID v4 format: 8-4-4-4-12 lowercase hex digits, e.g.
7a48756f-95b2-440c-9810-09a9ae60e452.

The :ID: property is compatible with Emacs org-id.el, which uses the same
property and format. Agents and Emacs can both manage IDs without conflict.
