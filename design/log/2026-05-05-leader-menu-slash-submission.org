* Context

leader-menu currently dispatches =command:= leader actions by
calling =ctx.ui.setEditorText(command)= and notifying the user to
press Enter. This is a two-step flow for every leader-bound slash
command (=/new=, =/compact=, =/reload=, =/copy=, =/tree=,
=/resume=, =/settings=, =/model=, …), which all of leader-menu's
shipped =defaults.json= local-menu entries are.

There is no public pi extension API to programmatically submit a
slash command. Built-in commands are dispatched only when the
default editor's =onSubmit= fires, inside a literal-text switch in
=interactive-mode.js= (=setupEditorSubmitHandler=). Extensions can
read =ctx.ui.getEditorComponent()= (added in pi 0.71.0) and call
=setEditorComponent()=, but the running editor instance is not
exposed.

The vim-mode extension (currently disabled, lives under
=agents/pi/disabled/vim-mode/=) sidesteps this by being a
=CustomEditor= subclass with its own =submitCommand(text)= helper
that calls =setEditorText(text); super.handleInput("\r")= — which
fires =onSubmit= and dispatches the command in one step. When
vim-mode was loaded, =command:= leader actions worked single-press;
with vim-mode disabled they fall back to the two-step
populate-and-press-Enter flow.

This change-record consolidates the submission capability in
leader-menu so it works regardless of whether vim-mode is loaded.
A thin =SubmitterEditor= base class lives in =extensions/lib/= so
both leader-menu and vim-mode reuse the same submission method
without a cross-extension import.

** Design decisions

- Single ownership :: Submission is leader-menu's responsibility
  end-to-end. The =submitCommand= method on vim-mode is removed;
  =VimEditor= inherits it from =SubmitterEditor= instead.
- Unconditional install :: leader-menu installs =SubmitterEditor=
  via =setEditorComponent()= at every =session_start=, regardless of
  what =getEditorComponent()= returns. This matches the user's "owns
  it entirely" framing and avoids a detect-and-yield branch that
  would silently regress to two-step in the presence of unrelated
  editor extensions. Trade-off: any other extension that calls
  =setEditorComponent()= without extending =SubmitterEditor= will
  clobber single-step submission. Documented as the contract.
- Shared base in =lib/= ::
  =SubmitterEditor extends CustomEditor= ships in
  =extensions/lib/editor.ts= (new module). Both leader-menu and
  vim-mode import it via the existing =../lib/= relative-import
  convention already used for =ansiPad= / =getExtensionName= /
  =registerLeaderMenu=. This avoids the cross-extension-import
  prohibition in =leader-menu/AGENTS.md= (=VIM_NORMAL_RESERVED_KEYS=
  is duplicated rather than imported from vim-mode) — =lib/= is a
  shared third party for both extensions, not a sibling.
- vim-mode wraps by structural inheritance, not runtime delegation
  :: =VimEditor= changes its parent class from =CustomEditor= to
  =SubmitterEditor=. This is the form of "wrap" that doesn't trip
  the prior plan's "do not compose invisible editor instances"
  rule (design/log/2026-05-01-keybindings-editor-composition.org):
  there is no second hidden instance with parallel
  text/render/autocomplete state — =SubmitterEditor= adds one method
  and zero state.
- Trivially stateless base :: =SubmitterEditor= must add no fields
  and override no methods other than to add =submitCommand=. Any
  drift toward statefulness re-introduces the state-skew hazard the
  prior plan rejected.
- Dispatch path :: leader-menu's =command:= action handler keeps
  =ctx.ui.setEditorText(command)= for visibility/parity with the
  pre-change behaviour, then calls
  =editorRef.submitCommand(command)= via a closure-captured
  reference held by the factory. No event bus indirection — both
  the factory and the action host live in the same extension.
- Ordering / collision behaviour :: When vim-mode is re-enabled
  later, vim-mode's =setEditorComponent()= runs *after*
  leader-menu's at =session_start= (load-order convention) and
  replaces leader-menu's instance with =VimEditor=. Single-step
  submission still works because =VimEditor= is a =SubmitterEditor=
  via inheritance — but leader-menu's closure-held =editorRef= is
  stale. Resolved by indirection: leader-menu reads the active
  instance from a module-scope registry in =lib/editor.ts= populated
  by the =SubmitterEditor= constructor, rather than from its own
  factory closure. The registry is module state shared across
  extensions because jiti-loaded modules with the same resolved path
  are deduplicated (verified empirically by =ansiPad= behaviour).
- Lifecycle :: At =session_shutdown=, leader-menu calls
  =setEditorComponent(undefined)= to restore pi's default editor.
  =SubmitterEditor= clears the active-instance registry on
  destruction.
- Documentation :: leader-menu README + AGENTS.md gain a "Slash
  command submission" section. vim-mode README + AGENTS.md note
  that submission lives in the shared base and =VimEditor= inherits
  it. The 2026-05-01 change-record is referenced from the new one
  to record that the "leader menu first, editor replacement second"
  default is being relaxed: leader-menu now installs an editor by
  default. The relaxation is justified by the trivially-stateless
  property, which the prior plan didn't have to consider because
  it was contemplating a full modal editor.

** Single-keystroke =command:= dispatch (interactive smoke test)

The plan's interactive smoke test (=ctrl+,= → =c= submits =/compact=
in one keystroke, etc.) requires a real terminal that speaks the
Kitty keyboard protocol or modifyOtherKeys (kitty, ghostty, foot,
wezterm, recent xterm). It cannot be reliably driven through tmux
=send-keys= because tmux strips Kitty-protocol bytes by default.
