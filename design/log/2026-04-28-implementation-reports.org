* Context

Tasks that warrant a formal plan get one: a file under =#+DEFAULT_PLAN_DIR=
linked from the parent task via =#+IMPORT:=, with =* Context=, =* Plan=, and
=* Implementation= sections (the plan skill defines this layout).  The
=* Implementation= section provides a durable, in-tree record of *what was
done* and *why*, separate from the git history.

Simple tasks \u2014 a one-line bug fix, a doc tweak, an upstream version bump \u2014
typically don't need formal planning.  Today they go directly =TODO \u2192 DONE=
with no record beyond the heading and the =CLOSED:= timestamp.  The git log
tells *what changed* but not *why* in any task-localised form, and there is
nothing for an agent or future contributor to read when resuming context.

The recent rollback of the keybindings capital-letter work is the
counter-example: that task *did* have a plan, so the four failed attempts
and what we learned from them now live in the plan's =* Implementation=
section.  For a plan-less task, that record would have nowhere to land.

This plan adds an *implementation report* convention: a minimal log file
under =#+DEFAULT_PLAN_DIR=, linked via =#+IMPORT:= just like a plan file,
but containing only an =* Implementation= section.  The pi tasks extension
prompts to scaffold one when a task is closed without a pre-existing
=#+IMPORT:=; the org-memory skill documents the convention so the agent
knows when to offer the prompt and what to write.

** Design decisions

- *Location and shape of the report file* :: Same directory and filename
  pattern as plan files (=#+DEFAULT_PLAN_DIR/YYYY-MM-DD-<slug>.org=,
  symlinked from the parent task via =#+IMPORT:=).  Same file headers
  (=#+TITLE=, =#+DATE=, =#+PARENT_ID=, =#+TODO=).  Difference is the
  body: only =* Implementation=, no =* Context= and no =* Plan=.  This
  symmetry means a report can be promoted to a plan later just by
  inserting the missing sections \u2014 no rename, no link rewrite.

- *When the prompt fires* :: On the status transition =TODO \u2192 DONE= or
  =STARTED \u2192 DONE= for a task that has no =#+IMPORT:= attached.
  =CANCELLED= does *not* trigger the prompt (no work was done; if the user
  wants a record of why, they can attach a report manually).  A user
  setting (default on) lets people opt out per project.

- *Who triggers the prompt* :: The pi tasks extension, when it observes
  the status cycle in its overlay.  Closures done outside the extension
  (Emacs status cycle, manual edit) do not auto-prompt; they get caught
  by the org-memory skill on the next agent interaction (the skill's
  resume protocol scans for closed tasks lacking =#+IMPORT:= and offers
  the report retroactively).

- *Minimum report content* :: Three bullet points: =Changed=, =Why=,
  =Follow-ups=.  Anything richer is opt-in.  The agent should populate
  these from the recent git history scoped to the task's =STARTED \u2192 DONE=
  window when possible, falling back to asking the user when the
  history is ambiguous.

- *Reuse the existing plan-create plumbing* :: The extension's plan
  scaffolding flow (path prompt, file creation, =#+IMPORT:= attachment,
  source-file save, agent message) already does almost everything.  The
  report flow is the same flow with a different scaffold body and a
  different agent prompt.  Avoid duplicating it; parameterise.

The agent prompt sent on report scaffolding should:

1. Identify the time window between the task's first =STARTED=
   transition (if observable) and =CLOSED:=.  When the window cannot
   be derived, fall back to "the most recent commits since the task
   was created".

   commit messages and any code context.

in =index.ts=; the agent reliably produces a draft for simple cases.

* Open questions

1. *Filename collision with promoted reports* :: If a report is later
   promoted to a plan, the file already has the agent-readable history
   in =* Implementation=.  Should the path prompt warn when a file
   already exists at the suggested path, or silently overwrite?  The
   existing plan flow appends to a =* Plan= section in pre-existing
   files; the report flow should probably do the analogous thing for
   =* Implementation= (append, don't overwrite).

2. *Auto-derivation of the time window* :: Is it worth tracking the
   first =STARTED= transition timestamp on the task itself (e.g. a new
   =:STARTED:= property) so the report's git-history scope is exact?
   Or is the heuristic ("recent commits") good enough for the use
   case?  Adding a property is cheap; mutating only on the first
   transition (not subsequent re-opens) keeps it stable.

3. *Whether to prompt for CANCELLED tasks* :: A cancellation note is
   often valuable ("we tried this, here's why we stopped").  Out of
   scope for this plan, but worth flagging as a candidate
   second-iteration feature.

4. *Should the skill's retroactive scan run on a TASKS.local.org
   selection switch?* :: The compact widget refreshes whenever
   selection changes; that's a natural moment for the scan to fire.
   Or is it strictly a session-start concern?  My instinct: session
   start only \u2014 mid-session offers would be intrusive.
