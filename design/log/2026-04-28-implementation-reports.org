* Context

Tasks that warrant up-front planning get a *change-record* file under
=#+DEFAULT_PLAN_DIR=, linked from the parent task via =#+IMPORT:=, with
=* Context=, =* Plan=, and =* Implementation= sections.  The file begins
life as a plan (the =* Plan= section drives the work) and becomes a
durable record once the work lands (=* Implementation= captures what
actually shipped, including failures rolled back).

Simple tasks \u2014 a one-line bug fix, a doc tweak, a version bump \u2014 don't
warrant up-front planning, so they currently go directly =TODO \u2192 DONE=
with no change-record at all.  The git log captures *what changed* but
not *why* in any task-localised form, and there is nothing for an agent
or future contributor to read when resuming context.  The recent
keybindings rollback is the counter-example: that task had a
change-record, so the four failed attempts and what we learned now live
in its =* Implementation= section.  For a plan-less task, that record
would have nowhere to land.

This plan adds a *retrospective flow* that produces the same
change-record artefact.  No new file shape, no new nomenclature: a
change-record is a change-record whether authored proactively (existing
flow, =p= keybinding in the tasks overlay) or retrospectively (new flow,
triggered by closing a task that has no =#+IMPORT:=).  The org-memory
skill is updated to describe the dual flow; the tasks extension is
updated to offer the retrospective scaffolding.

** Design decisions

- *Single artefact, two flows* :: The change-record file shape is
  unchanged from today's plan layout (=#+TITLE=, =#+DATE=, =#+PARENT_ID=,
  =#+TODO=, =* Context=, =* Plan=, =* Implementation=).  Whether the file
  is created before work begins or after work ends, the structure is
  identical.  In the proactive flow the agent helps draft =* Context=
  and =* Plan= up front; in the retrospective flow the agent drafts
  =* Context= and =* Implementation= from git history and
  conversation, and =* Plan= is either empty or contains a brief
  retrospective listing (with =DONE= / =CANCELLED= entries for the
  steps actually taken).  Either way, the file is referred to as a
  *change-record* in skill prose, and is linked from the parent task
  via =#+IMPORT:=.

- *Reuse the existing plumbing* :: =handlePlanEdit= in =index.ts=
  already does path-prompt + scaffold + =#+IMPORT:= attach + source
  save + agent prompt.  The retrospective flow is the same flow with
  a different agent-prompt body.  Parameterise the handler over a
  mode (=proactive= vs =retrospective=); leave the scaffold body
  identical.

- *When the retrospective prompt fires* :: On the status transition
  =TODO \u2192 DONE= or =STARTED \u2192 DONE= for a task that has no
  =#+IMPORT:= attached.  =CANCELLED= does not trigger.  A user setting
  (=tasks.changeRecordOnDone=, default =true=) lets people opt out per
  project.

- *Who triggers the prompt* :: The pi tasks extension only.  Closures
  done outside the extension (Emacs status cycle, manual edit) do not
  auto-prompt; they are caught by the org-memory skill on the next
  agent interaction (the skill's resume protocol scans for closed tasks
  lacking =#+IMPORT:= and offers a retrospective change-record).

- *No Emacs-side scaffolding command* :: The proactive flow stays
  reachable from Emacs via the existing =tasks-org-open-plan= (which
  already creates a plan-shaped change-record when none exists).  The
  retrospective flow is exclusive to the pi harness \u2014 there is no
  =tasks-org-create-report= equivalent.  The skill's retroactive scan
  covers Emacs-only users.

Replace any wording that distinguishes \"plan files\" from \"task files\"

=#+IMPORT:=.  Document that change-records are authored in either of

1. *Proactive*: scaffold up front, draft =* Context= + =* Plan=, then

   and =* Implementation= from git history and conversation; =* Plan=

   actually taken.

of truth for section conventions); this skill change is purely
documentation of the new flow and naming.

Acceptance: an agent reading the skill alone can decide which flow to
use for a given task and produces a change-record file that matches
the existing structure either way.

** TODO [#A] Tasks extension: scaffold change-record on DONE transition
:PROPERTIES:
:ID: c1d2e3f4-0003-4000-8000-300000000003
:END:
In =overlay.ts='s =cycleStatus()=, when the new status is =DONE= and
=task.importPath= is null, emit a new workflow request via a new
=onCreateChangeRecord= callback (mirroring =onEditPlan=).

In =index.ts=, route the request to the existing =handlePlanEdit=
flow with a new =mode: \"retrospective\"= argument.  The handler:

- Uses the same path prompt, the same scaffold body, the same
  =#+IMPORT:= attach, the same source save.
- Branches only on the agent prompt: =buildPlanDevelopmentPrompt=
  becomes =buildChangeRecordPrompt(mode)= and emits a different
  body for retrospective vs proactive (see task #5).

A user setting =tasks.changeRecordOnDone= controls whether the
prompt fires (default =true=); when =false=, status cycle to DONE
behaves as it does today.

The path-prompt overlay title can stay unchanged
(=New change-record for: <task summary>=) for both flows; the
distinction is conveyed by what the agent does next, not by the
prompt title.

Acceptance:
- Cycling a TODO/STARTED task to DONE in the overlay opens the path
  prompt; accepting it scaffolds the change-record file, attaches
  =#+IMPORT:=, and sends the retrospective agent prompt.
- Cycling a task that already has =#+IMPORT:= does *not* prompt.
- Cancelling the path prompt leaves the task DONE with no record.
- Cycling DONE -> CANCELLED does not prompt.

** TODO [#B] Skill resume protocol: detect un-recorded DONE tasks
:PROPERTIES:
:ID: c1d2e3f4-0005-4000-8000-300000000005
:END:
Update the =Starting or resuming work= section of the org-memory skill
so that when the agent loads task memory, it scans for DONE tasks that
have neither =#+IMPORT:= nor body content beyond the heading +
properties + =CLOSED:= line, and offers the user a retrospective
change-record for the most recent N such tasks.

Behaviour rules:
- The check is bounded: scan only =TASKS.org= roots and loaded included
  files, never =TASKS.archive.org=.
- The agent does not auto-create files; it offers, and only acts on
  user confirmation.
- The offer is made once per session at most; declining suppresses
  further offers in the session.

Acceptance: skill change is purely documentation; no code in the
extension is required because the agent itself performs the scan
through existing file-reading tools.

** TODO [#C] Retrospective agent-prompt template
:PROPERTIES:
:ID: c1d2e3f4-0007-4000-8000-300000000007
:END:
The agent prompt sent on retrospective scaffolding should:

1. Identify the time window between the task's first =STARTED=
   transition (if observable) and =CLOSED:=.  When the window cannot
   be derived, fall back to \"the most recent commits since the task
   was created\".
2. Run =git log --oneline= over that window scoped to files plausibly
   related to the task; OK to surface all recent commits when scoping
   is uncertain.
3. Draft the change-record's =* Context= (problem statement) and
   =* Implementation= (what shipped, including any rolled-back
   attempts) from the commit messages and code context.  =* Plan= may
   be left empty or filled with a retrospective bullet list of steps
   actually taken.
4. Show the draft to the user for approval before writing to disk.

Acceptance: prompt template lives alongside the existing
=buildPlanDevelopmentPrompt= as =buildChangeRecordPrompt(mode)=; the
agent reliably produces a usable retrospective draft for simple cases.

* Open questions

1. *Existing-file behaviour on the retrospective path* :: If the user
   accepts the path prompt and the file already exists at the
   suggested path, should the flow append to =* Implementation= (and
   leave =* Context= alone) or warn?  The proactive flow appends to
   =* Plan=; the symmetric behaviour for retrospective is to append to
   =* Implementation=.  My lean: yes, append; never overwrite.

2. *Track first =STARTED= timestamp as a property* :: Worth recording
   the first =STARTED= transition on the task (e.g. a new =:STARTED:=
   property, written by the extension on first transition only) so the
   retrospective git-history scope is exact?  Cheap to add; precise.
   My lean: yes, add it.

3. *Whether to prompt for CANCELLED tasks* :: Out of scope here, but a
   plausible second-iteration feature (\"we tried this, here's why we
   stopped\").  Defer.

4. *Retroactive-scan timing* :: Session-start only, or also when the
   selected task changes mid-session?  My lean: session start only \u2014
   mid-session offers would be intrusive.
