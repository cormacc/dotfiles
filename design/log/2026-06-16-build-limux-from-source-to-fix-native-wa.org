* Intent

Replace the prebuilt-binary =limux= derivation with a from-source Nix build —
Rust workspace plus the =am-will/ghostty= fork's =libghostty.so= built from one
coherent nixpkgs closure — so the native-Wayland EGL path renders embedded
ghostty surfaces correctly and the =GDK_BACKEND=x11= workaround can be dropped.
Keep the prebuilt derivation as =limux-bin= until native Wayland is verified on
this host.

* Summary

*Level:* production · *Tests:* smoke (=nix build= succeeds + native-Wayland renders on this host) · *Docs:* inline (maintenance/update-hash notes in this record)

The bet: the prebuilt =libghostty.so= =dlopen='s =libEGL/libGL/libwayland= by
bare name with no =DT_NEEDED=; spliced against nixpkgs GTK4/libwayland/Mesa +
the system NVIDIA EGL, the native-Wayland path renders blank. Building
libghostty + GTK4 + libwayland + Mesa from one nixpkgs closure should make the
stack coherent (Ghostty's own packaging docs recommend building from source
against system packages only). The make-or-break step is building the fork's
=libghostty.so= inside the Nix sandbox via a hash-pinned Zig-deps FOD — gated
first, before any Rust work.

** Scope
*** In scope
- New =pkgs/limux-ghostty/default.nix= building =libghostty.so= from the pinned fork via =zig_0_15= + a =zon2nix=-generated =deps.nix= FOD (=-Dapp-runtime=none=).
- Rewrite =pkgs/limux/default.nix= as a =rustPlatform.buildRustPackage= consuming =limux-ghostty=, replicating the =scripts/package.sh= install layout.
- Move the current prebuilt derivation to =pkgs/limux-bin/default.nix= (auto-exposed as =pkgs.limux-bin=).
- Preserve install layout: =bin/limux=, =libexec/limux/limux-host=, CLI, terminfo at =$out/share/limux/terminfo=, ghostty resources at =$out/share/limux/ghostty=.
- Preserve GL closure wiring (=libglvnd= + =addDriverRunpath.driverLink= on =LD_LIBRARY_PATH=).
- Verify native Wayland on this host (Sway, NVIDIA RTX 4070, driver 595); drop =GDK_BACKEND=x11= if it works.
- Inline maintenance notes: the per-release hash set (src, ghostty submodule rev, Zig-deps FOD, =cargoHash=).

*** Out of scope
- Upstreaming the derivation to nixpkgs.
- Substituting nixpkgs' =ghostty= 1.3.1 for the fork (ABI-incompatible with =limux-ghostty-sys=).
- Editing anything outside =pkgs/= and =TASKS.org=/this record.
- Deleting =pkgs/limux-bin= (the prebuilt fallback) — kept indefinitely as an escape hatch.

** Decisions
- Chose Approach A (two derivations: =limux-ghostty= Zig build + =limux= Rust build, with a committed =zon2nix=-generated =deps.nix=) ::
  only option giving a clean go/no-go spike gate (=nix build .#limux-ghostty=)
  before any Rust work; reuses the nixpkgs =ghostty= FOD pattern almost verbatim;
  keeps =cargoHash= and the Zig-deps hash in separate derivations.
  Rejected:
  - Approach B (single =stdenv.mkDerivation= doing Zig then Cargo): no clean spike gate; mixing the Zig FOD and Cargo vendoring in one derivation is fiddly; loses =rustPlatform= cargoHash/vendoring machinery.
  - Approach C (fetch a pre-generated =build.zig.zon.nix= from the fork like upstream's =update.nu=): the fork likely doesn't publish that file at its pinned rev, so =zon2nix= must run anyway — collapses into A.
- =limux-ghostty= is its own auto-exposed package file (not a private =let= binding) specifically so the spike gate is an independent =nix build= target.
- =limux= (source) becomes the default =pkgs.limux=; the prebuilt moves to =pkgs.limux-bin=. Both keep =mainProgram = "limux"=.
- nixpkgs prior-art = =pkgs/by-name/gh/ghostty/{package.nix,deps.nix,update.nu}=: =zig_0_15=, =dontSetZigDefaultFlags = true=, =zigBuildFlags = ["--system" "${deps}" "-Dcpu=baseline" "-Doptimize=ReleaseFast"]=, =deps = callPackage ./deps.nix=.

** Gotchas
- =rust/limux-ghostty-sys/build.rs= hardcodes =CARGO_MANIFEST_DIR/../../ghostty/zig-out/lib= and =.canonicalize().expect(...)= — it *panics* if =libghostty.so= is not at that exact relative path. The Rust derivation must stage the built lib there (or patch =build.rs= to read an env var in =postPatch=).
- =build.rs= also links =-lepoxy= and compiles =ghostty/vendor/glad/src/gl.c= from the ghostty *source* tree — so the Rust build needs both the =limux-ghostty= libghostty.so output *and* the ghostty source (=vendor/glad=), plus =libepoxy= as a build input.
- =.cargo/config.toml= hardcodes =-Wl,-rpath,/usr/local/lib/limux=; override/remove it (the wrapper uses =$out/lib=).
- =scripts/package.sh= produces ghostty resources via a *second* =zig build= invocation and copies =share/ghostty= + terminfo separately from the libghostty build — the resources are not a by-product of the =-Dapp-runtime=none= lib build.

** Risks
- Core bet may not hold :: one coherent closure might still not fix native-Wayland EGL on NVIDIA. Bounded: ISC-9 permits documented retention of =GDK_BACKEND=x11=; the work still ships a source build with =limux-bin= fallback.
- Maintenance burden :: per-release updates jump from 1 hash to ~4 (limux src, ghostty submodule rev, Zig-deps FOD, =cargoHash=), and the fork can drift. Accepted; mitigated by a maintenance-notes task and the indefinite =limux-bin= fallback.
- Spike may hard-block :: if the fork needs network codegen, vendored blobs, or a patched Zig the sandbox can't supply, Task 2 is a no-go. Explicit branch: keep =limux-bin= as default, record the no-go outcome — not a silent stall.
- Pinned-rev drift :: scout read the fork's =build.zig.zon= on =main=, not the exact rev limux v0.1.19's gitlink pins; the Zig version and deps set could differ. Task 1 resolves both from the pinned rev.
