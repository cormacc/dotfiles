* Context

Activating in-repo extension UIs through leader-menu currently uses
=command:= actions such as =command:/tasks= and =command:/diff toggle=.
leader-menu implements =command:= by writing the slash command into the
active editor and submitting it through =SubmitterEditor=. That is correct
for slash-command shortcuts, but undesirable for UI toggles when the user has
already typed text: the intermediate =setEditorText()= clobbers the input
area.

The fix is to make extension leader bindings dispatch events directly. Slash
commands remain available for typed use, but command handlers dispatch the
same event path as the keybinding so the implementation has a single entry
point.

Affected in-repo extensions for this bug: =tasks= and =git-diff=.

** Design decisions

- Event-dispatch for keybindings :: Leader-menu bindings should use bare event
  names, not =command:=, when the desired action is an extension UI action.
- Slash commands as compatibility shims :: =/tasks= and =/diff= remain, but
  their handlers emit the same events used by leader-menu bindings.
- Keep =command:= for actual slash-command shortcuts :: leader-menu's local
  built-ins like =/compact= still intentionally submit commands.

* Implementation

- Updated tasks leader-menu registrations to dispatch `tasks:show`,
  `tasks:new`, and `tasks:doctor` events.
- Refactored `/tasks` so typed slash commands dispatch those same events;
  event handlers run the existing overlay, task creation, and doctor logic.
- Updated git-diff leader-menu registrations to dispatch `git-diff:*` events.
- Refactored `/diff` so slash subcommands validate arguments and emit the same
  event names used by keybindings.
- Documented the event-dispatch convention in the tasks and git-diff READMEs.
