* Summary
Add Open WebUI to the =strix= NixOS profile as the LAN-facing chat UI for the co-hosted Lemonade OpenAI-compatible API, and use Lemonade's existing image-generation support before adding a separate image backend. Phase 1 enables the upstream =services.open-webui= module, points it at Lemonade, and exposes a single LAN-facing port. Phase 2 seeds Open WebUI's image settings against Lemonade's OpenAI-compatible image endpoint. Open WebUI is a frontend only; Lemonade remains the sole inference backend.

** Decisions
- Use Open WebUI: best fit for a lightweight local-LLM chat layer over Lemonade and available as =nixpkgs#open-webui= with an upstream =services.open-webui= NixOS module.
- Reject LibreChat (more platform-like, overbuilt for a domestic LAN) and LobeChat (Docker/full-stack, no nixpkgs package or NixOS service).
- Design toward =LAN -> Caddy -> Open WebUI -> Lemonade= per [[file:../home-network-infrastructure.org][home-network-infrastructure.org]]; Caddy/SSO is deferred.
- Keep the raw LAN Lemonade API reachable. Coding harness clients on other machines use it directly, so Open WebUI is an additional frontend, not the only LAN-facing path.
- Use Lemonade's =sd-cpp=-backed image generation as the first image backend (=hardware.amd-npu.enableImageGen=). Defer ComfyUI as a fallback if Lemonade's workflows prove insufficient (graph workflows, ControlNet, heavy LoRA experimentation).

** Gotchas
- =open-webui= is unfree; the repo already enables unfree packages.
- =services.open-webui= exposes =host=, =port=, =stateDir=, =environment=, =environmentFile=, =openFirewall=. It sets =StateDirectory = "open-webui"= and runs =open-webui serve --host ... --port ...=.
- =services.open-webui.environment= replaces the module default (=SCARF_NO_ANALYTICS=, =DO_NOT_TRACK=, =ANONYMIZED_TELEMETRY=) rather than merging — re-include them.
- Open WebUI persists config to SQLite under =stateDir= after first start. =environment= entries are first-run seeds only; later edits via env are ignored unless the admin UI changes them or state is reset.
- The module hard-codes =WEBUI_URL = "http://localhost:${cfg.port}"=, but merges =cfg.environment= after, so override is possible once Caddy is in front.
- =open-webui= is not in =nix-amd-ai='s Cachix; first build fetches ~301 MiB and builds the frontend (npm) and Python deps.
- Lemonade's canonical image endpoint path is =/api/v1/images/generations=, with =/v1/images/generations= as an OpenAI-client alias. Open WebUI's OpenAI image client uses the =/v1= form.
- ComfyUI is not currently packaged in nixpkgs (no =nixpkgs#comfyui= / =services.comfyui=). If needed, revisit via an external flake (e.g. =utensils/comfyui-nix=), a local wrapper, or an OCI container.

* Implementation
Background extracted to [[file:../home-network-infrastructure.org][design/home-network-infrastructure.org]]. Used the upstream =services.open-webui= module rather than hand-writing a systemd unit.

Phase 1 — in =hosts/strix/nixos-configuration.nix=:
- Added =openWebUIPort = 8080= next to =lemonadePort=.
- Kept Lemonade bound to =0.0.0.0= with its firewall hole open (LAN coding-harness clients hit it directly).
- Enabled =services.open-webui= with =host = "0.0.0.0"=, =port = openWebUIPort=, =openFirewall = true=.
- Seeded the OpenAI provider: =OPENAI_API_BASE_URL = "http://127.0.0.1:13305/v1"=, placeholder =OPENAI_API_KEY = "sk-local-lemonade"=, =ENABLE_OLLAMA_API = "False"=.
- Re-included the module's telemetry-off defaults.
- Added =systemd.services.open-webui.after/wants = [ "lemond.service" ]= (nix-amd-ai names the unit =lemond=).

Phase 2 — image generation, same file:
- =ENABLE_IMAGE_GENERATION = "True"=, =IMAGE_GENERATION_ENGINE = "openai"=.
- =IMAGES_OPENAI_API_BASE_URL = "http://127.0.0.1:13305/v1"=, =IMAGES_OPENAI_API_KEY = "sk-local-lemonade"=.
- =IMAGE_GENERATION_MODEL = "SD-Turbo"= (auto-downloaded by Lemonade on first use), =IMAGE_SIZE = "512x512"=.
- =IMAGES_OPENAI_PARAMS= as JSON =({steps=4, cfg_scale=1.0})=, matching Lemonade's SD-Turbo example. Note: Open WebUI 0.9.5 reads the env var =IMAGES_OPENAI_PARAMS= but persists it under the config key =IMAGES_OPENAI_API_PARAMS=.

First-run / recovery notes:
- State lives under the module default =/var/lib/open-webui= (data under =data/=).
- The first browser sign-up becomes the admin account; restrict signup in the admin UI afterwards.
- Because Open WebUI persists provider/auth settings on first start, later env changes are ignored. Fix via the admin UI or stop the service and reset =/var/lib/open-webui=.
- If =SD-Turbo= hasn't been downloaded yet, expect a long first image request while Lemonade fetches it.

Diagnostic notes (captured after first end-to-end run surfaced a 400):
- *Lemonade image-model ids are case-sensitive.* =GET /v1/models= is authoritative; the registry exposes =SD-Turbo= and =SDXL-Turbo=. The admin UI persists whatever string the admin types (typos included) into =/var/lib/open-webui/data/webui.db=, where it then shadows =IMAGE_GENERATION_MODEL=. Symptom: Lemonade returns =404 Model not found: <typo>=, but Open WebUI's =routers/images.py= wraps that into =HTTPException(status_code=400, detail="[ERROR: Not Found]")= before returning it to the browser.
- *Diagnostic shortcut.* When the UI shows an opaque =400: [ERROR: Not Found]=, =journalctl -u lemond= reveals the real underlying status (e.g. =Error 404: POST /v1/images/generations= preceded by =Failed to load image model: Model not found: <id>=). =journalctl -u open-webui= shows the wrapper and exact base URL hit.
- *Probing without auth.* =/api/v1/images/config= is auth-gated, but Lemonade's =/v1/models= and =/v1/images/generations= are open on the LAN and accept the full OpenAI payload shape — a direct =curl= against Lemonade quickly rules out backend/payload issues and isolates the bug to Open WebUI's persisted state.

* Validation
- Phase 1 runtime on =strix=: rebuilt successfully, =systemctl status open-webui= healthy, LAN browser loaded Open WebUI, Lemonade-provided text models listed and completed a chat request, raw Lemonade port still reachable for coding-harness clients.
- Phase 2 end-to-end on =strix= [2026-05-21]: direct =POST http://strix.internal:13305/v1/images/generations= with =model=SD-Turbo= returns =HTTP 200= + base64 PNG. Initial 400 from Open WebUI traced via =journalctl -u lemond= (=Model not found: SD-turbo= → 404) and =journalctl -u open-webui= (=routers/images.py= wraps 404 → 400) to a model-name typo persisted in =/var/lib/open-webui/data/webui.db=; fixed in =Admin Panel → Settings → Images=. Image generation from the chat UI then succeeds end-to-end.
