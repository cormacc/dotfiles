* Summary

Add a `strix` NixOS host profile for the Framework Desktop / AMD Ryzen AI Max+ 395 that follows the xps15 split model: NixOS is built as `.#strix`, Home Manager remains a separate `.#default` apply, and shell aliases continue to derive from `/etc/hostname` / `cfgName`. The host should reuse the workstation+sway stack, avoid NVIDIA modules, enable the AMD AI/Lemonade stack from `noamsto/nix-amd-ai`, and ship a bootstrap script for applying the profile on top of a fresh NixOS live-CD install. Because this work is being planned away from the new machine, generated hardware config must be deferred to bootstrap time rather than guessed locally.

** Decisions
- Keep `nix-amd-ai` pinned to its own nixpkgs input; upstream warns not to make it follow this flake's `nixpkgs` because that defeats its binary cache and triggers backend rebuilds.
- Keep Home Manager separate from `nixosConfigurations.strix`; bootstrap should run both `nixos-rebuild` and `home-manager switch` rather than embedding Home Manager as a NixOS module.
- Start with the existing workstation profile (`nixos-workstation.nix`) and remove only incompatible host-specific modules such as NVIDIA.
- Expose Lemonade on the LAN for other devices and enable image generation from the initial profile.

* Context

`nix-amd-ai` provides the NixOS module for `hardware.amd-npu`, Lemonade, XRT/XDNA, FastFlowLM, ROCm, and Vulkan backends. The module requires kernel >= 6.14, an XDNA 2 NPU, and `video`/`render` user groups; this repo's workstation kernel and `nixos-base.nix` user groups already line up. Its Cachix cache (`https://nix-amd-ai.cachix.org`) should be added at both flake/nixConfig and system-level nix settings so the large inference packages do not rebuild locally.

* Implementation

Tactical outcomes worth preserving beyond the plan-task acceptance criteria:

- Kernel ≥ 6.14 (required by =amdxdna=) is satisfied automatically by =nixos-workstation.nix= pinning =linux_6_18=. No strix-specific kernel pin needed; this also means the assertion is unchecked locally and will only trip on the target if the workstation kernel pin ever regresses below 6.14.
- =nix-amd-ai= Cachix is wired in two places: =flake.nix= =nixConfig= (flake-eval time, already-trusted users) **and** =hosts/strix/nixos-configuration.nix= =nix.settings= (post-activation, fresh-install path). Kept host-scoped rather than added to =nixos-base.nix= so unrelated hosts don't carry an unused substituter.
- The placeholder =hosts/strix/hardware-configuration.nix= uses =throw= rather than a stub so any accidental rebuild fails loudly with the actionable bootstrap message. Side-effect: =nix flake check --impure= will fail for =nixosConfigurations.strix= on this machine until bootstrap runs on the target — acceptable per the plan's "pre-hardware validation" trade-off.
- =bootstrap.sh= replaces a 2-line stub left over from the "all in on nix" refactor commit (=9538e5d=). The new script is generalised across hosts; future profiles can opt in by shipping a placeholder =hosts/<host>/hardware-configuration.nix= and running =./bootstrap.sh <host>=.

** DECIDED Should the bootstrap script be host-specific or generic, and where does it live?
:PROPERTIES:
:DECIDED: [2026-05-19 Tue]
:END:
Decision (raised by user during plan review): make =bootstrap.sh= generic and place it at the repo root. The script takes the profile name as its required CLI argument (=./bootstrap.sh strix=) and parameterises every step (host directory, hardware-file copy, =nixos-rebuild --flake .#<profile>=, Home Manager config). Detection of an unbootstrapped hardware file uses the literal =BOOTSTRAP_PLACEHOLDER= sentinel, which is also generic. Future hosts (=t580=, =t470p=, etc.) can adopt the same idiom by shipping a placeholder =hosts/<host>/hardware-configuration.nix= and running =./bootstrap.sh <host>= on a fresh install.
