* Context
The org-memory protocol uses a reserved =:selected:= tag on a task heading to
mark the contributor's currently-active task. The tag is read by the pi tasks
extension (compact widget / overlay), the tasks-org Emacs package
(=tasks-org-toggle-selected=), and any agent following the org-memory skill.

The tag lives inside the same =TASKS.org= and plan files that are version
controlled and shared between contributors. That has three concrete problems:

1. *Guaranteed merge conflicts*: Two contributors setting selection at the same
   time produce a textual conflict on the same heading line. There is no good
   semantic merge — the result is incoherent regardless of which side wins.
2. *Personal state in shared history*: =git log= ends up tracking
   "who selected what when," which is noise and can leak working patterns.
3. *Cross-file scan cost*: The current Emacs implementation
   (=tasks-org--clear-selected-everywhere=) walks every plan file referenced
   from =TASKS.org= to clear stale selections before applying a new one.

The fix is to move selection out of shared content into a gitignored
=TASKS.local.org= file, expressed as a single top-level keyword:

#+begin_src org
#+SELECTED: 2b87d48c-d158-44ff-8fea-87c99b570144
#+end_src

The file holds no task structure. Selection becomes an ID lookup against the
existing task graph rather than a tag-mutation operation. Absent file or
missing keyword means "no selection."

** Design decisions

- Single-keyword schema :: =TASKS.local.org= contains only =#+SELECTED: <UUID>=
  at v1. Other local state (personal priority overrides, scratch annotations)
  is out of scope; if needed, propose schema extensions in a follow-up plan.
- ID lookup, not tag mutation :: Both readers and writers operate on a UUID
  pointer in the local file. The selected heading is resolved by =:ID:= match
  against =TASKS.org= and the plan files it references via =:PLAN:=. No
  =:selected:= tag is written anywhere.
- Backwards-compatible read path :: On read, if =TASKS.local.org= is missing or
  has no =#+SELECTED:=, fall back to scanning for an existing =:selected:= tag
  so a half-migrated repo still functions. Writers always migrate: they remove
  any stale =:selected:= tags they encounter from shared files and write the
  new pointer to the local file.
- Gitignored, not gitattributes-merge :: =TASKS.local.org= is added to
  =.gitignore=. We do not try to keep it in the index with a custom merge
  driver — the whole point is to remove this state from version control.
- File-watch, not direct IPC :: pi continues to refresh from disk via its
  existing file watcher; the watcher is extended to monitor
  =TASKS.local.org=. No new live protocol between Emacs and pi.
- Skill documentation is part of v1 :: The org-memory skill currently lists
  =:selected:= as the reserved operational tag. That documentation must change
  in the same workstream so agents picking up the protocol after this change
  do not regress to writing the tag.

* Open questions

Resolve before or during implementation; record outcomes in *Implementation*.

- Keyword name :: =#+SELECTED:= is the chosen name. Confirm there is no
  collision with org-mode's reserved keywords or any existing project keyword
  before locking it in. (Quick sanity check: grep org sources / docs and
  =rg "^#\\+SELECTED" .= across the repo.)
- Write atomicity :: =TASKS.local.org= is small but read concurrently by the
  pi watcher. Decide whether to write via =rename(2)= for atomic replacement
  (recommended) or accept transient empty-file reads.
- Multiple working copies :: A contributor with multiple checkouts of the same
  repo will have independent =TASKS.local.org= files. Confirm this is the
  desired behaviour (per-checkout selection) rather than per-user state stored
  outside the repo (e.g. =$XDG_STATE_HOME/pi/<repo-id>/selected=).
- Indicator scope :: If the optional Emacs indicator (subtask
  =2a6c8e1f=) is implemented, decide whether it lives in =tasks-org-mode= by
  default or behind a customisation flag. Default-on is more discoverable;
  default-off avoids surprising users who don't use selection.
- Skill renaming consequences :: The recent rename of =:PLAN:= → =:INCLUDE:=
  in the org-memory skill has not been applied to =TASKS.org= or the existing
  plan file. Decide whether to bundle that cleanup with this workstream or
  keep it strictly out of scope. Default: out of scope; track separately.
