* Context
The selected workstream is `Refine tasks extension`, covering UX and persistence refinements for the pi tasks extension under `agents/pi/extensions/tasks/`.

Current implementation snapshot:
- `index.ts` owns extension lifecycle, `/tasks`, keybinding suggestions, file watchers, linked plan loading, compact selected-task widget, plan creation, task creation, and archiving.
- `overlay.ts` owns the expanded split-pane task tree, status cycling, selection, collapse state, new-task shortcuts, archive shortcut, details pane, and file saves.
- `parser.ts` parses and serializes org TODO task trees while preserving non-task org content and `:PLAN:` link forms.
- `README.md` documents the current commands, compact selected-task widget, linked plans, selection, and archive behaviour.

** Design decisions
- Preserve the plain-org protocol :: All changes should keep `TASKS.org` and linked plan files useful without pi, including UUID `:ID:` properties and clickable `[[file:...]]` plan links.
- Prefer one task UI owner :: The compact selected-task widget and expanded `/tasks` view should share the same load/refresh flow rather than stacking independent persistent overlays.
- Avoid masking the input flow :: Workflows that require prompts or confirmations must either run inside the expanded task UI or temporarily collapse/close it before invoking `ctx.ui.input` / `ctx.ui.confirm`.
- Backfill IDs only in the loaded task graph :: Auto-ID insertion should touch `TASKS.org` and linked plan files loaded through `:PLAN:` properties, not arbitrary org files.
- Sequence to avoid rework :: The UI-owner merge changes the owner of collapse state, plan-link rendering, and the compact selected-task surface. Do the merge before those polish workstreams, and treat masking-prevention as choosing the compact-UI placement after the merge lands.

* Implementation
Design and decision log — populated as workstreams are executed.

Sequencing rationale:
- ID auto-insert is independent and low-risk; do first.
- Overlay merge is the architectural foundation; the prompt-flow fix, collapse defaults, plan-link display, and compact-UI placement all assume the merged controller exists.
- Documentation is a single pass at the end so user-visible behaviour is consolidated in one diff.

Per-task validation gate:
- LSP diagnostics should be clean aside from the repository's current module-resolution limitations in this standalone checkout.
- `/reload` in pi should be run interactively before closing final validation.
- The subtask's listed verification cases should pass.

Decisions recorded:
- Default collapse rules (`2003536d`) :: Collapse completed subtrees everywhere by default, but always keep the selected path visible. With no selection, show top-level tasks only. With a selection, expand the selected path and collapse sibling subtrees.
- Compact selected-task UI placement (`5d2ae654`) :: Use a reserved above-editor widget, not an overlay. This reserves layout space and avoids masking scrollback. The compact view is capped at 6 lines.
- Prompt strategy (`11a58aed`) :: Temporarily close/hide the expanded tasks UI and use normal `ctx.ui.input` / `ctx.ui.confirm` dialogs, then refresh and reopen for create/archive flows.
- ID backfill preservation :: Preserve org semantics and unchanged non-task content; do not promise strict byte-for-byte preservation of rewritten task subtrees.
