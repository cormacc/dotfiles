* Context

The =tasks= extension and the =org-tasks= / =org-plan= skills currently
treat =TASKS.org= as a self-contained per-project task graph.  Day-job
work for the team lives in Jira at
https://your-org.atlassian.net.  The two universes don't talk: a
contributor pulling a Jira ticket has to retype the title and lose the
Jira ID; a contributor closing a task has to context switch to the
browser to update the corresponding ticket.

The Atlassian MCP server (=atlassian= in pi's MCP config) was wired up
recently and exposes ~39 tools covering Jira issues, transitions,
comments, JQL search and worklogs against cloud
=00000000-0000-4000-8000-000000000000=.  Available scopes are
=read:jira-work= and =write:jira-work=.  Auth is delegated to the MCP
server, so the extension and skills can stay credential-free.

We expect Jira surfaces (search, my open issues, cross-task views) to
grow beyond per-task linkage, so the integration is built as a /new/
=jira= pi extension and a new =org-jira= skill (extending =org-tasks=)
rather than absorbed into the existing =tasks= extension.  =org-tasks=
and =org-plan= stay tracker-agnostic; they only gain a documented
extension-point convention.

The =tasks= extension itself gains /generic/ external-issue support
(property name =:LINKED_ISSUES:=, file keyword =#+ISSUE_URL_BASE:=)
that is intentionally tracker-agnostic: =tasks= renders badges and
opens browser links for any issue tracker the user configures.  All
Jira-specific knowledge — MCP routing, workflow names, slash
commands — lives in the =jira= extension and =org-jira= skill, so
=tasks= stays useful for users who don't have the =jira= extension
installed (badges still render, =J= still opens the browser).

The change-record for this work covers the umbrella task and all four
nested subtasks; each is rendered as one or more concrete plan items
under =* Plan= so they can be picked up incrementally.

** Design decisions

- /New =jira= pi extension; =tasks= and the existing skills stay
  tracker-agnostic./  Decision (2026-04-28): a separate extension is
  worth the extra surface because Jira use is expected to grow beyond
  per-task linkage.
  - =jira= pi extension owns slash commands (=/jira clone=, =/jira
    claim=, =/jira comment=, =/jira create=), MCP-driven workflows,
    and any Jira-only UI.
  - =org-jira= skill (new) owns the Jira-aware authoring conventions
    (which subset of =:LINKED_ISSUES:= tokens map to Jira; what
    =#+JIRA_*= keywords mean) and the agent prompts that drive MCP
    calls.  It /extends/ =org-tasks= rather than living inside it.
  - =tasks= extension gains generic =:LINKED_ISSUES:= rendering and
    URL composition, but holds /no/ Jira knowledge.
  - =org-tasks= / =org-plan= skills must not mention Jira.  They
    document the extension-point convention (unknown =#+= keywords
    and drawer properties round-trip cleanly) and nothing more.

- /Generic external-issue linkage in =tasks=./  A multi-valued
  =:LINKED_ISSUES:= drawer property holds whitespace-separated tokens.
  Each token is either:
  1. A bare key, e.g. =MBFW-123=, resolved against =#+ISSUE_URL_BASE=
     (see below); rendered as the key itself.
  2. A full org link =[[URL][LABEL]]=, resolved directly; rendered as
     =LABEL=.
  The two forms can mix freely on the same line:
  #+begin_example
  :PROPERTIES:
  :ID: 01234567-89ab-…
  :LINKED_ISSUES: MBFW-123 MBE-45 [[https://github.com/foo/bar/issues/42][gh#42]]
  :END:
  #+end_example
  Tag form (=:MBFW-123:= on the heading) was rejected to avoid
  polluting the semantic =:tag:= namespace.

- /URL template via =#+ISSUE_URL_BASE=, owned by =tasks=./  Add to
  =TASKS.org= preamble:
  #+begin_example
  #+ISSUE_URL_BASE: https://your-org.atlassian.net/browse/{ID}
  #+end_example
  Resolution rule for bare keys:
  1. If the template contains the substring =[[{ID}]]=, replace it
     with the key (after URL-encoding).
  2. Otherwise treat the template as a prefix and append the key.
  This means both =https://example/browse/{ID}= and
  =https://example/browse/= work for the common cases, and
  unusual URL shapes (=https://issues.example.com/?id={ID}&view=full=)
  are also expressible.  =TASKS.local.org= may override the keyword
  (last-write-wins, mirroring =#+SELECTED:=).  When the keyword is
  absent, bare keys still render but =J= refuses with a notification
  pointing to the missing keyword.

- /=jira= owns its own =#+= keywords, scoped to MCP concerns./
  =#+JIRA_CLOUDID:=, =#+JIRA_PROJECT:=, =#+JIRA_BASE_URL:= are read
  /only/ by =jira= and =org-jira=.  =tasks= never sees them.  Their
  purposes:
  - =#+JIRA_CLOUDID= — avoid an
    =atlassian_getAccessibleAtlassianResources= round-trip per call.
  - =#+JIRA_PROJECT= — disambiguate short keys (=123=) and seed
    =/jira create=.
  - =#+JIRA_BASE_URL= — host-prefix used by =jira= to identify
    which =:LINKED_ISSUES:= tokens are "its own" when running
    =/jira claim=, =/jira comment=, etc.

- /Atlassian MCP, not direct REST./  All Jira access goes through the
  existing =atlassian= MCP server, exclusively driven by the agent.
  The =jira= extension itself stays I/O-free; slash commands draft a
  prompt, the agent performs MCP calls, and results are written back
  into org files via the existing tasks-extension scaffolding.

- /Two-tier UX: read-only properties + agent-driven workflows./
  - Read-only display (=:LINKED_ISSUES:= badges, =J= browser-open) is
    handled by =tasks= directly — no MCP needed for the happy
    path; works without =jira= installed.
  - Workflows that touch Jira (=clone=, =claim=, =comment=, =create=,
    =transition=) are exposed by =jira= as slash commands that
    /draft an agent prompt/ rather than executing themselves.  This
    keeps extension code small, lets the user review the action, and
    works the same way as the existing change-record retrospective
    flow.

- /=jira= writes bare keys, not org links, into =:LINKED_ISSUES:=./
  When =/jira clone= or =/jira create= mutates the property, it
  appends or sets bare keys (=MBFW-123=).  The org-link form is for
  manual entries to non-templatable trackers.  This keeps the common
  case compact and lets =/jira= work without any link rendering of
  its own.

- /=:LINKED_ISSUES:= is created on first use, never backfilled./
  Existing tasks are not migrated.  New tasks created by the =tasks=
  extension (=/tasks new=, =n=, =N=) do not pre-create the property.
  The drawer entry only appears when:
  1. A user manually adds it.
  2. =/jira clone= seeds a new task with the cloned issue's key.
  3. =/jira create= appends the new key after promoting an existing
     task to a Jira issue.
  Mirrors the existing =:STARTED:= behaviour (only written on first
  transition out of =TODO=).  Keeps =TASKS.org= compact for tasks
  that have no external issues.

- /Subtasks are independent issues, not Jira sub-tasks./  We deliberately
  /do not/ try to mirror the org parent/child hierarchy onto Jira
  Epic/Story/Sub-task hierarchy.  Each task carries its own (possibly
  empty) =:LINKED_ISSUES:= list; the relationship between them is a
  human-readable one expressed in the org tree.  A future task can add
  Epic-link awareness if there's demand.

- /Confluence is out of scope./  The MCP server also exposes Confluence
  tools.  Noted as a future follow-up (=#+CONFLUENCE_*= keywords could
  mirror the Jira keywords, or Confluence pages can ride on
  =:LINKED_ISSUES:= via the org-link form) but not implemented here.

- /Sandbox is =SAND=./  All write-path smoke tests during stages B and
  C run against the =SAND= project.  No write-path call hits any other
  project until the relevant stage is signed off.

- /Graceful degradation when MCP is offline./  Display of cached
  =:LINKED_ISSUES:= and browser-link navigation work unconditionally
  (they live in =tasks= and don't touch the MCP).  Anything that
  needs the MCP surfaces a clear notification ("Atlassian MCP not
  connected — run /mcp reconnect atlassian") rather than failing
  silently.

- /Question-handling mode./  Minor implementation ambiguities batch
  into =* Open questions= for end-of-stage review.  Anything that
  affects downstream design (data shape, public extension API,
  cross-extension contract) is raised immediately.
