* Context
The org-memory/task-memory tooling currently lives inside this dotfiles repo under `agents/`. Generic, harness-agnostic skills are symlinked into `~/.agents/skills`, while pi-specific extensions and companion skills are symlinked into `~/.pi/agent/`. This makes local development pleasant because files can be edited in place and pi can hot-reload extensions, but it does not provide a clean reproducible packaging boundary for sharing the protocol and extensions with other users.

The selected approach is to add an in-repo dual-mode package around the org-memory subset of the existing layout first. The development mode keeps the current out-of-store symlinks. A store/package mode exposes the org-memory resources through a Nix package and Home Manager wiring. The same layout should also be consumable as a pi package via a package manifest, so extraction to a dedicated repository later is mostly a repository move rather than a redesign.

Initial package scope is deliberately narrow: generic skills `org-tasks`, `org-plan`, and `org-jira`; pi extensions `tasks`, `jira`, `leader-menu`, and `emacsclient` (the latter is a real loadable pi extension that `tasks` reaches into for its Emacs server helper, so packaging the bundle implies packaging the whole extension); and runtime helper modules those extensions import (`pi/extensions/lib/{pi-utils,editor,wm}.ts`). Other generic skills, pi-specific skills, prompts, settings, and unrelated extensions remain local dotfiles resources for now.

** Design decisions
- Keep `agents/skills` generic :: The packaged skills remain harness-agnostic and continue to target `~/.agents/skills` for pi, Claude Code, Codex, and other agents that honor that location.
- Keep pi-specific resources under `agents/pi` :: The packaged pi extensions and any future companion skills stay separate from generic skills and continue to target `~/.pi/agent/skills` and `~/.pi/agent/extensions`.
- Package only the org-memory slice first :: The first package boundary is `org-tasks`, `org-plan`, `org-jira`, `tasks`, `jira`, `leader-menu`, and required helper modules. This avoids accidentally publishing unrelated personal workflow extensions.
- Editable mode is for the dotfiles owner, packaged mode is for collaborators :: Editable mode keeps the existing whole-directory `mkOutOfStoreSymlink` wiring and continues to install the entire local suite (every skill and extension in the checkout), because that is the local-dev workflow. Packaged mode uses per-entry symlinks from a Nix store package and exposes only the explicit org-memory resource set, so collaborators get exactly the documented bundle and nothing else.
- Do not package user settings :: `agents/pi/settings.json` and user override files such as `tasks-ext.json`, `jira-ext.json`, and `leader-menu.json` are out of scope and remain local configuration. The package exposes resources, not opinionated global pi settings.
- Add pi package metadata without forcing pi-native installation :: A package manifest lets pi install/load the bundle directly, but Nix/Home Manager remains the primary install path for this repo.
- Defer repository extraction :: Extraction is valuable for public/team sharing, but first proving the package boundary in-place reduces migration risk.

** Spike: pi package manifest format
:PROPERTIES:
:DATE: [2026-05-06 Wed 21:59]

Reference: pi 0.73.0 docs `docs/packages.md`, `docs/extensions.md`, `docs/skills.md`, plus extension loader at `dist/core/extensions/loader.js`.

*** Manifest shape
A pi package is an npm-style directory whose `package.json` carries an optional `pi` key declaring resource roots:

#+begin_src json
{
  "name": "@cormacc/agent-org-memory",
  "version": "0.1.0",
  "description": "Org-memory task protocol skills + pi extensions",
  "keywords": ["pi-package"],
  "pi": {
    "extensions": ["./extensions"],
    "skills": ["./skills"]
  },
  "peerDependencies": {
    "@mariozechner/pi-coding-agent": "*",
    "@mariozechner/pi-tui": "*",
    "@sinclair/typebox": "*"
  }
}
#+end_src

Key rules confirmed from docs and pi source:
- `pi.extensions` is an array of paths/globs (relative to package root). For each entry pi loads:
  1. direct files `*.ts` / `*.js`,
  2. subdirectories containing `index.ts` / `index.js`,
  3. subdirectories containing their own `package.json` with a `pi` field.
  Subdirectories that have none of these (e.g. `extensions/lib/`) are silently skipped, so helper-only directories cohabit safely with extensions.
- `pi.skills` is an array of paths/globs. Skills are directories containing a `SKILL.md`; pi recurses to find them.
- Globs support `!exclude` patterns and consumer-side filtering (`+path` / `-path`) in `settings.json`. Not needed for the first cut, but available if the slice grows.
- Without a `pi` manifest, pi auto-discovers from convention dirs `extensions/`, `skills/`, `prompts/`, `themes/`. Equivalent to specifying those dirs explicitly. We will declare the manifest explicitly so the slice is documented in-tree.
- Add the `pi-package` keyword for gallery discoverability. `pi.video` / `pi.image` are gallery-only metadata; skip for now.

*** Dependencies
Import audit of the four packaged extensions plus `lib/` (excluding tests) shows the only external runtime imports are `@mariozechner/pi-coding-agent`, `@mariozechner/pi-tui`, `@sinclair/typebox`, and `node:*` builtins. Per pi docs these three packages are bundled by pi itself and *must* be declared as `peerDependencies` with `"*"` (never `dependencies`, never bundled). No `dependencies` or `bundledDependencies` are needed for the org-memory bundle.

Test-time imports (`vitest` etc.) do not appear in the package output — tests are excluded by the Nix package per the boundary task.

*** Install paths for collaborators
Pi accepts three install sources, all viable for the bundle:
1. =pi install npm:@cormacc/agent-org-memory= — the standard route once published. Requires npm publish.
2. =pi install git:github.com/cormacc/dotfiles= (or a future extracted repo) — clones, runs `npm install`, then resolves the `pi` manifest. Works without npm publish.
3. =pi install /path/to/store= — local path entry written to `settings.json` without copying. This is what Home Manager's packaged mode produces: the Nix store path of the built package is added to global pi settings (`~/.pi/agent/settings.json`), or per-entry symlinks under `~/.pi/agent/extensions/` and `~/.agents/skills/` reference the same store paths directly. Pi treats the latter as ordinary auto-discovered locations, so the `pi` manifest is not strictly required for the symlink form — but we ship it anyway so route (1)/(2) work for downstream consumers.

Resource auto-discovery rules (`docs/skills.md`, `docs/extensions.md`) mean per-entry symlinks under the standard locations work without a manifest:
- `~/.pi/agent/extensions/<name>/index.ts` is auto-loaded.
- `~/.agents/skills/<name>/SKILL.md` is auto-loaded.
- Helper directories without `index.ts` (`lib/`) are skipped by the extension scanner.

Package tree to ship (matches the source layout under `agents/`, so the Nix package is a near-trivial copy + filter and `pi install /path/to/agents` works directly):

*** Packaged mode
Bug discovered and fixed during validation:
- pi's extension loader resolves `../lib/…` imports from the
  *symlink path* (`~/.pi/agent/extensions/<ext>/index.ts`), not from
  the realpath. Without a sibling `lib/` entry under the same
  parent, all four packaged extensions fail to load with
  `Cannot find module '../lib/…'` errors.
- Fix: packaged mode now also creates
  `~/.pi/agent/extensions/lib` as a symlink to
  `${cfg.package}/pi/extensions/lib`. The lib directory has no
  `index.ts` and no `package.json`, so pi's extension scanner
  silently skips it as a candidate extension while keeping it
  reachable for sibling-import resolution.
- The change-record's earlier `Decisions falling out of the spike`
  entry asserting that no `lib/` symlink was needed is therefore
  stale; the corrected behaviour is captured here.

building the same per-entry symlinks the Home Manager module would
