* Context

Agent / pi configuration is layered: a *global* tier lives in this
dotfiles repo (=AGENTS.md=, =agents/skills/=, =agents/pi/=) and is
symlinked into =~/.agents= and =~/.pi/agent= via [[file:agents.nix]];
a *project-local* tier may live in any repo (a project-level
=AGENTS.md=, project-scoped skills, project-specific pi extensions or
conventions). Friction with either tier is most visible while
working *in some project*, but the durable fix has to land at the
right tier — global friction is fixed in dotfiles, project-local
friction is fixed in that project.

We want a single skill that captures the feedback at the moment it
arises and routes it to the right tier as actionable TODO work,
without forcing a context-switch out of whatever the contributor is
currently doing.

** Design decisions

- Single skill, repo-agnostic :: One =self-improvement= skill that
  is available in every session (lives under =agents/skills/= so it
  symlinks into =~/.agents/skills/=). It does not key its behaviour
  off "am I inside dotfiles?" — instead it keys off *what the
  feedback is about*.
- Routing by target tier :: When the skill is invoked it
  classifies the affected artefact and routes accordingly:
  - *Project-local target* (project =AGENTS.md=, project-scoped
    skill / extension / convention, project tooling) → file the TODO
    in the *current project's* =TASKS.org= under =* Agent feedback=.
  - *Global target* (dotfiles-managed =AGENTS.md=, =~/.agents/skills=,
    =~/.pi/agent= contents) → file the TODO in
    =~/dotfiles/TASKS.org= under =* Agent feedback=.
  - *Ambiguous* → ask the user to disambiguate before routing; if
    they decline, default to the current project (least disruptive)
    and tag with =:tier-unknown:= so it can be re-routed later.
- Submit + triage as one flow :: There is no separate "client" and
  "server". Whichever session the skill is invoked in *also* runs
  the triage logic for entries it routes locally. For entries
  routed to dotfiles, the originating session hands off via
  pi-intercom and the receiving dotfiles session does the triage
  there.
- Local-tier flow (no intercom) :: For project-local entries the
  skill writes directly to the current project's =TASKS.org= via
  =tasks_insert_task=. No transport needed; the originating session
  is already in the right place to act on it.
- Cross-tier flow via pi-intercom :: For dotfiles-tier entries the
  skill discovers a live dotfiles session via =intercom action:list=,
  spawns one if none is alive (e.g. =pi= in a tmux session named
  =dotfiles-feedback= with cwd =$HOME/dotfiles=), and sends a
  =intercom action:send= envelope (fire-and-forget). The dotfiles
  session triages the inbound message, inserts the TODO, and replies
  asynchronously with the new task UUID and a "plan now or
  backlog?" prompt that lands in the originating session's inbox.
- Persistence in =TASKS.org= under =* Agent feedback= :: At both
  tiers, feedback becomes durable as TODO entries under a dedicated
  =* Agent feedback= section. Inserted via =tasks_insert_task= so
  =:ID:=, =:CREATED:=, and tags follow protocol. From there the entry
  can grow a change-record like any other task. Avoids inventing a
  parallel inbox; existing tooling (selection, archiving,
  =org-plan=) applies unchanged.
- Free-form payload + auto-detected metadata :: Sender supplies a
  free-form description; the skill auto-attaches origin session
  name, =cwd=, git remote, current branch, timestamp, sender type
  (=human= vs =agent=), and an optional transcript snippet. The
  affected-target classification is derived during triage rather
  than required at submission, keeping the submit UX one-shot.
- Confirmation policy by sender type :: When the sender is the
  *agent itself* (proposed in response to recurring missteps or
  user corrections) triage auto-inserts the drafted task without
  further prompting. When the sender is a *human* triage surfaces
  its drafted summary/body for confirmation before inserting,
  because the human's intent is being reformulated and they
  should sign off on the wording.
- Triage classifies + dedupes :: Before inserting, triage tags the
  entry with the affected target (e.g. =:skill_org-tasks:=,
  =:ext_jira:=, =:agents-md:=, =:project-convention:=) and searches
  existing open entries in =* Agent feedback= for near-duplicates,
  merging evidence into the existing entry rather than creating a
  parallel TODO.

** Out of scope

- A general "lessons learned" inbox separate from =TASKS.org=.
- Auto-editing =AGENTS.md= / skill files without going through a
  =TASKS.org= entry + change-record + the usual review flow.
- Cross-machine delivery (pi-intercom is local-only). Multi-host
  feedback can be addressed later if the need arises.
- Auto-rerouting an entry that was filed at the wrong tier — the
  skill flags ambiguous cases at submission time but does not move
  entries between repos after the fact.

** Notes / decisions taken during implementation

- /Free-form text envelope, not JSON/. The cross-tier message
  is a plain string with a =[self-improvement]= prefix tag and
  a structured =Origin:= block. Rationale: pi-intercom messages
  are agent-to-agent prose; using the prefix avoids inventing a
  schema while still letting the dotfiles agent recognise the
  message.
- /Acknowledgement uses =send=, not =reply=/. The originating
  session's submit call was =send= (fire-and-forget), so there
  is no in-flight =ask= for the dotfiles side to =reply= to.
  Triage emits a fresh =send= with the =[self-improvement]=
  prefix; the originating-session agent treats inbound messages
  with that prefix as the planning prompt.
- /Confirmation policy/. Inverted from the usual "trust
  humans more" default: human-sender → confirm wording (because
  the dotfiles-side agent has reformulated their words);
  agent-sender → auto-insert (because there's no human at the
  sender end to confirm anything).
- /No archiving of feedback entries here/. Once an entry is
  filed under =* Agent feedback=, it follows the standard
  =org-tasks= status discipline; archiving on DONE/CANCELLED is
  the existing protocol's job, not this skill's.
