* Context

Pi 0.71.0 added =ctx.ui.getEditorComponent()= ([[https://github.com/badlogic/pi-mono/issues/3935][issue #3935]]) so editor-replacing
extensions can /wrap/ a previously installed editor instead of
clobbering it. =setEditorComponent()= remains single-owner — last
writer wins, the previous editor's container is cleared — so any two
extensions that both replace the editor silently disable one of them
(same single-owner pattern as #3071, #3541, #3536).

Our =keybindings= extension installs =VimEditor= unconditionally from
=session_start= ([[file:../../agents/pi/extensions/keybindings/index.ts]]
around line 2438):

#+begin_src typescript
ctx.ui.setEditorComponent(
  (tui, theme, keybindings) => {
    const editor = new VimEditor(tui, theme, keybindings);
    …
    return editor;
  },
);
#+end_src

It does not read =getEditorComponent()=, and =VimEditor='s constructor
has no =base= parameter. So today we silently lose to (or silently
disable) any other editor extension — e.g. =pi-vim=,
=@jordyvd/pi-image-attachments= — depending on load order.

There is no current in-tree collision: no other extension under
=agents/pi/extensions/= calls =setEditorComponent=. This is
future-proofing rather than a live bug. It is worth doing because
=keybindings= is the editor extension we expect to combine with
third-party editor extensions.

The pi binary on this machine is 0.71.0 (=/nix/store/...-pi-0.71.0=),
so =getEditorComponent()= is available at runtime. Older pi installs
do not have the API, so the call must be guarded.

** Design decisions

- Do not compose invisible editor instances :: The first draft planned
  to instantiate the previous editor factory and delegate unhandled
  Insert-mode keys to that editor. Final review rejected this: the
  wrapped editor would keep its own invisible text/render/autocomplete
  state while =VimEditor= remained the returned component. That risks
  lost edits, duplicated app-keybinding handling, and hard-to-debug
  state skew.
- Leader menu first, editor replacement second :: The primary feature
  is the which-key-style leader menu. Since modal editing is disabled
  by default, the default path should not call =setEditorComponent()=
  at all. Use upstream's extension shortcut + custom overlay APIs for
  insert-only leader menus, and reserve custom-editor replacement for
  actual modal editing.
- Upstream API patterns :: Follow the linked =extensions.md= examples:
  use =pi.registerShortcut()= for global shortcuts, =ctx.ui.custom(...,
  { overlay: true })= for temporary key-capturing UI, =ctx.ui.setWidget=
  only for persistent/simple widgets, and =ctx.ui.setStatus()= for
  footer state. Follow =modal-editor.ts= / =rainbow-editor.ts= by
  keeping =VimEditor= self-contained and using =super.handleInput()=
  for normal editor fallback rather than attempting to layer two full
  editor instances.
- Modal install policy :: On =session_start=, install =VimEditor= only
  when persisted settings have ={ modal: true }=. In insert-only mode,
  preserve pi's default editor or any third-party editor already
  installed by another extension.
- Runtime mode switching :: Keep the existing =keybindings:set-mode-vim=
  and =keybindings:set-mode-emacs= events. Switching to vim mode should
  install =VimEditor= and remember the previous editor factory when
  =getEditorComponent()= is available. Switching back should restore
  that remembered factory when possible; otherwise pass =undefined= to
  restore pi's default editor.
- Conflict handling :: If vim mode is enabled while another editor
  factory is already present, warn the user that modal mode replaces
  the current editor. Do not attempt unsafe input delegation. The
  =getEditorComponent()= API is still useful for detection and restore,
  even if we avoid composition for now.
- Mode/status rendering :: Prefer =ctx.ui.setStatus("keybindings", …)=
  for =INSERT=/=NORMAL=/=VISUAL= feedback over appending mode text to
  the editor border. This uses the documented status/footer API and
  reduces editor rendering coupling. Keep border rendering only if
  status proves insufficient during implementation.
- Action dispatch :: Extract leader action dispatch away from
  =VimEditor=. Command and event actions should work from both overlay
  and modal paths. Existing =passthrough:= actions should be replaced
  with explicit action handlers where public APIs exist; any remaining
  unsupported passthrough should fail with a clear notification rather
  than synthesizing raw keys into an invisible editor.
- Out of scope :: True editor middleware/chaining remains out of
  scope until there is a real second editor extension to support and
  upstream exposes a safer composition contract than read-then-replace.

* Implementation

- 2026-05-01 final review revised the plan before implementation.
  The original read-then-wrap idea was replaced with a simpler design:
  keep leader menus independent of editor replacement, use upstream
  =registerShortcut= and overlay APIs for insert-only mode, and install
  =VimEditor= only when modal editing is actually enabled.
- Implemented a =LeaderActionHost= abstraction so menu construction is
  independent of =VimEditor=. The default action host delegates to the
  active modal editor when one exists; otherwise it uses public UI/API
  calls where available or inserts slash commands for manual Enter.
- Added =LeaderMenuOverlay= for insert-only =alt+space= / =alt+,=
  navigation via =ctx.ui.custom(..., { overlay: true })=. Default insert-only session
  startup now leaves the current editor untouched.
- Vim mode now remembers =ctx.ui.getEditorComponent?.()= before
  replacement and restores that factory (or pi default editor) when
  switching back to emacs mode.
- Mode feedback moved to =ctx.ui.setStatus("keybindings", …)=. The
  editor border now only shows pending count/operator/leader state.
- Follow-up after interactive verification: =alt+m= conflicted with
  pi-intercom's registered shortcut. Replaced the single prefix with
  direct root shortcuts: =alt+space= for the Space leader and =alt+,=
  for the comma leader. Vim Insert mode uses the same direct roots.
- Verification: =bun build agents/pi/extensions/keybindings/index.ts
  --target=node --external @mariozechner/pi-coding-agent --external
  @mariozechner/pi-tui= succeeds.
- User confirmed the updated implementation works interactively after
  replacing =alt+m= with =alt+space= / =alt+,=.
