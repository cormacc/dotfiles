Review Summary: farra/dev-agent-backlog

Captured 2026-05-01. Source: https://github.com/farra/dev-agent-backlog

Executive take: dev-agent-backlog is close in spirit to the local org-memory system: plain-text org, agent-readable, design-first, and commit-friendly. The main philosophical difference is that their design docs are canonical while backlog.org is a transient working surface. Our system keeps TASKS.org as the canonical task graph and links durable change-records via #+IMPORT:.

What they do well

  1. Explicit design-doc lifecycle: #+STATUS: Draft | Review | Accepted | Active | Complete | Archived, with review commands to resolve questions and finalize tasks.
  2. OPEN / DECIDED questions: questions are parseable task-like headings rather than loose prose.
  3. Handoff notes: :HANDOFF: gives the next session a concise “start here” note.
  4. Completion summaries: distinguishes mid-work progress from final shipped summary.
  5. Reconciliation / doctor commands: checks stale links, stale entries, broken source links, and status inconsistencies.
  6. Ready/dependency model: draft multi-agent protocol includes :CLAIMED_BY:, :BLOCKED_BY:, and :WAITS_FOR:.
  7. Progress isolation: per-task progress files reduce merge conflicts in multi-agent workflows.
  8. Attribution: optional :WORKED_BY: and :COMPLETED_BY: metadata.

Where the local system is stronger

  • No duplicated canonical task state: #+IMPORT: avoids the checkout/reconcile drift inherent in their model.
  • Better lifecycle/audit mechanics: UUID :ID:, :CREATED:, :STARTED:, CLOSED:, append-only :LOGBOOK:, parent propagation, and archiving.
  • Better local/private state: TASKS.local.org handles selection and local drafts cleanly.
  • Better tracker integration shape: generic linked issues plus companion tracker skills is less coupled than one property per tracker.

Ideas worth incorporating

  1. Add :HANDOFF: support and surface it in resume / selected-task UI.
  2. Structure * Open questions with OPEN / DECIDED states.
  3. Add optional #+STATUS: to change-records: Draft, Review, Accepted, Active, Complete.
  4. Add a /tasks doctor health-check command for broken imports, duplicate UUIDs, stale status, invalid blockers, and missing timestamps.
  5. Add ready-task query support using WAITING, :BLOCKED-BY: task:<UUID>, and possibly :WAITS_FOR:.
  6. Prompt for completion summary when closing top-level tasks.
  7. Add optional acceptance criteria / Definition of Done conventions for non-trivial tasks.
  8. Consider solution/lesson capture as separate knowledge-base material, not mixed into task lifecycle.

Ideas to avoid

  • Replacing TASKS.org + #+IMPORT: with a transient checkout/reconcile backlog.org.
  • Replacing UUIDs with human-readable IDs, though aliases could be useful later.
  • Claude-specific task semantics in the core protocol.
  • Transcript links by default.
  • Bidirectional sync with external systems; link and re-fetch when needed is healthier.

Bottom line

The third-party system validates the direction of the local org-memory protocol, but our core model is more robust. The best borrowings are around review lifecycle, handoff, question resolution, readiness/dependency queries, and health-check tooling—not the central backlog checkout model.
