* Context
The selected task asks for a review of the org-memory protocol — primarily the
`org-tasks` and `org-plan` skills plus the tasks extension documentation — as a
cross-session memory mechanism for agents, not only as a human task list.

A review sub-agent was run against:

- `agents/skills/org-tasks/SKILL.md`
- `agents/skills/org-plan/SKILL.md`
- `agents/pi/extensions/tasks/README.md`
- `TASKS.org`

The review found material protocol gaps that can make agent memory ambiguous,
unsafe, or difficult to reconstruct across sessions:

- High: level-2 plan tasks are documented as reusing TASKS.org subtask UUIDs
  while the original subtasks remain in TASKS.org, creating duplicate `:ID:`
  values in the loaded task graph and ambiguous selection/resume/update targets.
- Medium: retrospective records rely on `:STARTED:` and `CLOSED:` timestamps,
  but direct `TODO -> DONE` completion may produce no `:STARTED:` lower bound.
- Medium: `CLOSED:` preservation is underspecified for reopened tasks, so later
  closures may keep stale close times and distort git-log scoping / archive sort.
- Medium: `#+IMPORT:` and plan-path prompts have no documented trust boundary;
  arbitrary absolute paths, parent traversal, or symlink escapes could cause
  unexpected file reads/writes.
- Medium: `#+ISSUE_URL_BASE:` plus the `J` opener does not document scheme
  restrictions, so local overrides could launch non-web URL handlers.
- Low: existing tasks do not consistently have bracketed `:CREATED:` values,
  despite the protocol making them required.

As an agent-memory review, these findings point at a broader principle: durable
memory needs unambiguous identity, deterministic lifecycle timestamps, explicit
trust boundaries for loaded content, and migration/compatibility rules for
legacy task data.

** Design decisions
- Treat duplicate IDs as the top priority :: Agent resume and parent-status
  propagation depend on UUID identity. If an ID can name two physical nodes,
  every higher-level memory operation becomes unreliable.
- Subtask migration is the intended identity model :: When a task gets a
  change-record, its existing subtasks are moved into the plan with their
  `:ID:` values intact and are deleted from the parent `TASKS.org` subtree.
  There should be no parser coalescing or mirror-ID indirection for that flow.
- Specify lifecycle semantics before changing tooling :: The skills are the
  durable contract. Extension and Emacs behaviour should be aligned after the
  protocol defines direct-close and reopen semantics.
- Make trust boundaries part of the protocol :: Cross-session memory imports
  are effectively code-agent input. Import paths need explicit project-boundary
  rules. Linked issue URLs, however, are project-local configuration inside the
  repository trust boundary, so non-HTTPS schemes should remain allowed as long
  as opening is implemented without shell interpolation.
- Prefer compatibility rules over silent backfill :: Existing task files already
  violate the ideal `:CREATED:` shape. Document legacy tolerance and offer an
  explicit normalization command rather than rewriting history opportunistically.
