* Context
The selected task is to review the third-party org-based memory system at
https://github.com/farra/dev-agent-backlog and identify ideas worth adapting for
the local org-memory system (`org-tasks`, `org-plan`, `org-jira`, and the
associated pi extensions).

A first-pass review was completed on 2026-05-01. The durable HTML summary is
saved at [[file:2026-05-01-dev-agent-backlog-review-summary.html][dev-agent-backlog review summary]].

** First-pass conclusion
`dev-agent-backlog` is closely aligned with this system's goals: org-mode,
plain-text, agent-readable, design-first task memory. Its central model differs:
design docs are canonical and `backlog.org` is a transient checkout/reconcile
working surface. Our current model keeps `TASKS.org` as the canonical task graph
and links durable change-records via `#+IMPORT:`.

The recommendation from the first-pass review is to keep our core model and
adapt selected ideas around handoff, question state, review lifecycle,
ready/dependency queries, completion summaries, and health-check tooling.

Deliverable for this change-record: the agreed *decisions* plus the
implementation work that flows from them. The change-record was originally
scoped review-only and was recast on 2026-05-02 to absorb the three
implementation follow-ups (SKILL updates, multi-valued `:BLOCKED-BY:` parser
work, `/tasks doctor`) so the spec and the work that delivers it stay in
one place. Independent reviews spawned by this work — e.g. the
self-improvement skill review — remain as their own top-level tasks in
`TASKS.org`.

** Design decisions to revisit
- Keep canonical imported task graph :: Do not replace `TASKS.org` and
  `#+IMPORT:` with a copied checkout/reconcile `backlog.org` model.
- Prefer generic extension points :: Continue using tracker-agnostic
  `:LINKED_ISSUES:` and companion tracker skills rather than typed core
  properties for every external system.
- Add structured review affordances incrementally :: Handoff notes,
  `OPEN`/`DECIDED` questions, change-record status, and doctor/readiness
  commands can be adopted without destabilizing the existing protocol.
- Retain `:BLOCKED-BY:` for dependency/blocker metadata :: Do not add a
  separate `:WAITS_FOR:` fan-in property. Instead, allow `:BLOCKED-BY:` to
  carry one or more blocker references so the same field covers both simple
  blockers and multi-prerequisite readiness gates.

** DECIDED What delimiter convention should multi-valued `:BLOCKED-BY:` use?
Decision [2026-05-02 Sat 11:51]: use org's native `:NAME+:` continuation-line
syntax. The first blocker stays on `:BLOCKED-BY:`; additional blockers each
appear on their own `:BLOCKED-BY+:` line within the same `:PROPERTIES:`
drawer. Rationale:
- Backward-compatible with all existing single-value usage.
- Avoids comma-vs-free-text ambiguity in `human:` blocker references and
  in URLs that may contain commas.
- Each blocker is independently editable, addable, and removable without
  string surgery on a delimited list.
- Native org syntax → round-trips cleanly through generic org parsers.
Protocol/parser implications are tracked under task
`60e4cb26-583b-4d07-b04c-6e2efdbec410`.
