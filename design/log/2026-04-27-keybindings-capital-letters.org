* Context

The keybindings extension's leader-menu dispatch uses raw string equality
to match incoming key data against configured key strings:

#+begin_example
// handleLeader — current
const match = node.children?.find((c) => c.key === data);

// mergeChildren — current
const idx = existing.findIndex((c) => c.key === child.key);
#+end_example

Raw equality works when both sides agree on the representation.  It
fails when one side uses a normalised key name and the other uses the
raw terminal byte:

- Terminal sends capital A as the single byte =0x41= = the string ="A"=.
- =matchesKey(data, "A")= returns =true=.
- =matchesKey(data, "shift+a")= also returns =true= — =matchesKey= treats
  the two forms as equivalent.
- But ="shift+a" === "A"= is =false=, so a config entry written as
  =key: "shift+a"= will never fire when the user presses =A=.

The symptom is =SPC t A= silently exiting the leader without running an
action when the config entry for that slot was written in the
=shift+letter= normalised form rather than the bare capital-letter form.

The same mismatch affects =mergeChildren=: a config with =key: "A"= and
a suggested config with =key: "shift+a"= for the same slot are not
recognised as clashing, so both entries end up in the tree and only the
first one ever fires.

A third affected site is =warnClashingLeaderKeys=, which checks whether
a trigger key conflicts with a vim normal-mode binding by looking it up
in the =NORMAL_MODE_KEYS= set.  The set contains bare capitals (="A"=,
="I"= etc.), so a trigger specified as ="shift+a"= silently misses the
warning.

** Design decisions

- Use =matchesKey= for all comparisons :: Replacing raw equality with
  =matchesKey(incoming, configured)= (or =matchesKey(a, b)= for
  symmetric clash detection) makes every comparison consistent with the
  rest of the extension, which already uses =matchesKey= everywhere else.
  No new normalisation helper is needed.

- Canonical key name in the tree :: After matching with =matchesKey=, the
  winning entry's =key= field retains whatever string was in the config.
  This is intentional: the canonical form is preserved for the overlay
  display; only the =match= step needs to be lenient.

- =warnClashingLeaderKeys= :: Change the NORMAL_MODE_KEYS lookup to use
  =matchesKey= so that =key: "shift+a"= correctly warns about clashing
  with the vim =A= (insert-at-end-of-line) binding.

Four implementation attempts on top of the plan commit
=20e439901c2b0a5972097548e183daa0576e0bb4= were all rolled back without
landing because none of them changed the observed user-facing behaviour:
=SPC t A= still failed to fire its bound action.  The user reported that
the binding only fires after pressing =shift=, releasing it, and then
pressing the lower-case letter \u2014 a sequence the analysis below could
never explain from first principles.

** What we know after these attempts

- The three matching strategies are individually correct in isolation:
  - Legacy ="A"=: =canonicalizeInput("A") === "shift+a"= and
    =matchesKey("A", "shift+a")= both return true.
  - Kitty ESC[97;2u: =parseKey= decodes to ="shift+a"= which equals the
    stored key.
- =parseKey("A")= returns the raw character ="A"= verbatim, *not*
  ="shift+a"=, because pi-tui's catch-all for printable ASCII at the
  bottom of =parseKey= returns =data= unchanged.  This is the trap that
  invalidated the matchesKey-only approach.
- Files at =~/.pi/agent/extensions/keybindings/= are byte-identical to
  the dotfiles repo source after each commit, so the "stale code" theory
  is ruled out at the file-system level.
- We did not confirm whether pi/Bun has a runtime module cache that
  survives =/reload= or session restart.  This remains the most likely
  reason the user's observation that "behaviour is unchanged" persisted
  across structurally different fixes.
- The user's specific symptom \u2014 "the binding fires only after shift is
  pressed, released, and the lower-case letter is pressed" \u2014 cannot be
  explained by any of the comparison logic we inspected.  It would
  require either:
  - some upstream layer (pi-tui, pi core, or the terminal) to be
    accumulating modifier state across discrete key events, or
  - the user's terminal sending a sequence that we haven't seen and
    therefore haven't designed for.

** Recommended next steps

The user has chosen to roll back to lower-case-only bindings rather
than continue chasing this.  The simplest implementation that works
across every terminal is:

- Use only lower-case letters as leader-menu keys.
- Where capital-letter overlay shortcuts already exist (=A=, =P=, =U=
  in the tasks overlay), keep them inside the overlay and do *not* try
  to surface them at the leader-menu level.
- If a capital-letter binding is genuinely needed in future, first
  reproduce the failure with =TASKS_KB_DEBUG=1= so the actual raw byte
  sequence is visible \u2014 *then* design the fix against real input.

** Resolution (2026-04-28) \u2014 misdiagnosis

Task cancelled. The keybindings extension was never the bug.

While adding parallel debug logging to the tasks overlay's =handleInput=
(so we could see what the overlay was actually receiving when the leader
fired through to it), we found the overlay calling:

#+begin_example
if (matchesKey(data, "A")) { ... }   // archive
if (matchesKey(data, "P")) { ... }   // publish
if (matchesKey(data, "U")) { ... }   // unpublish
if (matchesKey(data, "N")) { ... }   // new-child
if (matchesKey(data, "J")) { ... }   // open-linked-issues
#+end_example

=matchesKey= expects the canonical name (=shift+a=, =shift+p=,
=shift+u=, =shift+n=, =shift+j=) for shifted letters, not the literal
uppercase character. Fixing the five call sites in
=agents/pi/extensions/tasks/overlay.ts= resolved the original symptom.

Lessons:
- The four rolled-back commits all chased the wrong layer. Every change
  was internally consistent and correct; none addressed the actual
  consumer of the dispatched key.
- The =TASKS_KB_DEBUG=1= notification approach mooted above turned out
  to be the right diagnostic instinct \u2014 but it needed to live in the
  *consumer* (overlay) too, not only in the leader.
- A no-op fix at the leader level looked like "behaviour unchanged"
  precisely because the leader was already forwarding the key correctly;
  the overlay was silently rejecting it.
