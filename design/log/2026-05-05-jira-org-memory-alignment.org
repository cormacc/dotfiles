This draft plan extracts the Jira-specific follow-up work that was
originally mixed into the broader org-memory review change-record
(=design/log/2026-05-01-org-memory-agent-memory-review.org=,
parent task =724c0332=). The generic org-memory task should focus on
shared task/plan protocol mechanics; Jira-specific integration checks
can be considered separately.

The prior org-memory implementation pass already landed some Jira
alignment work:

- =agents/skills/org-jira/SKILL.md= no longer describes Jira-derived
  plan tasks as UUID mirrors; Jira keys remain external references in
  =:LINKED_ISSUES:=.
- =org-jira= and the Jira README document Jira project/base-url
  configuration as project-local trusted configuration.
- =org-tasks=' agent-memory guidance cross-links Jira planning/resume
  context so agents know to re-fetch upstream Jira state when resuming
  a linked task.
- =jira_clone_apply= delegates insertion to =tasks_insert_task=,
  which now applies the project-root sandbox at the primitive layer.

Remaining Jira work is mostly validation and policy detail: reopen /
multi-transition semantics for auto-transition, clone-specific
sandbox tests, and a Jira-linked resume scenario test. This record is
Draft until that scope is accepted.

** Design decisions

- Keep Jira-specific checks out of the generic org-memory review ::
  The selected org-memory task still tracks shared protocol work
  (LOGBOOK, import sandboxing, agent-memory tests). Jira integration
  has extra external-state and MCP semantics, so it deserves its own
  review plan.
- Treat prior shipped Jira docs/code as context, not active work ::
  This plan does not rewrite the earlier commits. It records what
  already landed and focuses the executable tasks on remaining gaps.
- Prefer offline regression coverage :: Later implementation should
  stub parsed Jira fields and status events; it should not depend on a
  live Atlassian MCP except for explicit smoke tests.
